# 6스탯 등장인물 생성기 시스템

## 원문/요지

사용자는 주인공과 등장인물의 일관성을 위해 6개 코어 스탯 기반 캐릭터 시스템을 제안했다. 이 시스템을 적용하면 캐릭터의 성격, 행동, 이벤트 조건, 게임 내 판정이 더 일관될 수 있고, LLM이 새로운 등장인물을 생성할 때도 같은 세계관과 형식을 유지하기 쉬워진다.

핵심은 직접 찍는 스탯을 6개로 제한하고, 나머지 인상값과 기능은 태그/파생값/상태값으로 처리하는 것이다.

```text
직접 찍는 코어 스탯:
감각, 사회, 이성, 자아, 충동, 신체
```

## 핵심

이 아이디어는 `Escape from the Office`의 캐릭터를 “회사 안에서 실제로 있을 법한 사람”으로 유지하면서도, 게임 시스템에 바로 연결할 수 있는 구조를 만든다.

중요한 원칙:

- 기본 스탯 총합은 60이다.
- 초기 캐릭터의 각 스탯은 0~20 범위다.
- 특별 요청이 없으면 최소값은 3 이상이다.
- 21~30은 아이템, 성장, 오염, 특수 상태, 회사 시스템 개입 등으로만 가능하다.
- 높은 스탯은 대체로 이롭다.
- 낮은 스탯은 강한 결핍으로 작동한다.
- 매력, 속도, 위압감, 신뢰감은 코어 스탯이 아니라 파생 인상값이다.
- 체형, 외형, 복장, 직책, 상태는 숫자가 아니라 태그로 표현한다.

## 기대 효과

- 캐릭터 성격과 행동 패턴을 스탯에 맞춰 일관되게 유지할 수 있다.
- 대화, 설득, 거짓말, 조사, 도주, 이상현상 반응을 이벤트 조건으로 연결하기 쉽다.
- LLM 생성기가 캐릭터를 만들 때 세계관, 출력 형식, 톤을 유지하기 쉽다.
- “현실 오피스 스릴러 / 블랙코미디 / 미세한 이상현상” 장르를 벗어나지 않도록 안전장치가 된다.
- NPC를 단순 묘사가 아니라 게임 기능, 비밀, 첫 등장 장면까지 가진 콘텐츠 단위로 만들 수 있다.

## 구현 후보

### 1. 설계 문서화

`docs/design/Character_Stats_and_Generator.md`에 다음 내용을 정리했다.

- 6개 코어 스탯 정의
- 수치 범위와 총합 규칙
- 높은/낮은 스탯의 밸런스 철학
- 체형/외형/복장/직책 태그
- 파생 인상값
- 캐릭터 출력 형식
- 생성 방식과 금지 방향
- 생성기 프롬프트 계약 후보
- 예시 캐릭터 2명 요약

### 2. LLM 생성기 스크립트 후보

나중에 구현한다면 다음 형태가 가능하다.

- 입력: 부서, 성별/나이대, 높은/낮은 스탯, 역할, 비밀 강도, 이상현상 노출 여부
- 출력: Markdown 캐릭터 시트
- 선택 출력: YAML/JSON 캐릭터 데이터
- 검증: 6스탯 총합 60, 초기 최대 20, 직접 스탯 6개 외 금지

### 3. 게임 시스템 연결 후보

- 감각: 단서 발견, 위화감 감지, 거짓말 징후 발견
- 사회: 설득, 소문, 사내정치, 책임 회피
- 이성: 문서 분석, 시스템 규칙 해석, 숫자/패턴 판독
- 자아: 공포/세뇌/기억 오염 저항
- 충동: 빠른 선택, 도주, 기습 행동, 망설임 여부
- 신체: 피로 저항, 도주/추격, 부상 저항, 물리 행동

## 주의점

- 이 아이디어는 아직 후보이며, 기존 플레이어 스탯/상태 시스템을 즉시 대체하라는 지시가 아니다.
- 기존 상태값(체력, 정신력, 배터리, 허기, 갈증)과 기존 판정 능력치(논리, 공감, 의지, 침착, 인터페이스, 신체)가 있다면 통합/매핑 결정이 필요하다.
- LLM 생성기는 캐릭터를 과장된 괴물이나 판타지 직업처럼 만들면 안 된다.
- 높은 스탯에 큰 벌칙을 붙이면 안 된다.
- 공포는 괴물/악마가 아니라 사무실의 불편함, 기록 오류, 침묵, 반복 업무로 보여줘야 한다.
- 예시 캐릭터 2명은 좋은 기준이지만, 실제 콘텐츠로 승격할지는 별도 결정이 필요하다.

## 설계자에게 물어볼 질문

- 이 6스탯 시스템을 플레이어와 NPC 모두에 적용할 것인가, NPC 생성 전용으로 둘 것인가?
- 기존 판정 능력치와 통합한다면 어떤 매핑이 자연스러운가?
  - 예: 이성 ↔ 논리/인터페이스, 사회 ↔ 공감, 자아 ↔ 의지/침착, 신체 ↔ 신체
- 기존 자원 상태값(체력, 정신력, 배터리, 허기, 갈증)은 코어 스탯과 별개로 유지할 것인가?
- LLM 생성기 출력은 Markdown 문서가 충분한가, 아니면 런타임 데이터용 YAML/JSON 스키마가 필요한가?
- 파생 인상값은 실제 판정 수치로 쓸 것인가, 아니면 대사/연출용 레이블로만 쓸 것인가?
- 박태경, 윤혜린 예시 캐릭터를 실제 NPC 후보로 `docs/content`나 데이터 파일에 승격할 것인가?
