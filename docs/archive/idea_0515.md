# idea_0515.md

`escape from the office` 컨셉 아이디어 정리본.

이 문서는 지금까지 나온 아이디어를 하나의 방향성으로 묶은 초안이다.  
목표는 세부 설정집이 아니라, 이후 `DESIGN.md`, `UI_RULES.md`, `STORY_FLOW.md`, `HORROR_IDEAS.md`로 분리할 수 있는 **분위기 기둥**을 만드는 것이다.

---

## 1. 핵심 정체성

이 게임은 단순한 회사 배경 텍스트 어드벤처가 아니다.

> 이상한 사내 프로그램을 실행한 것 같은 TUI 호러 어드벤처.

플레이어는 게임을 한다기보다, 사내 서버나 내부망에 접속해서 잘못된 로그, 문서, 시스템 메시지, 생존자 기록을 훔쳐보는 느낌을 받아야 한다.

핵심 감각은 다음이다.

> 터미널 자체가 믿을 수 없는 공간이 된다.

그리고 반복해서 플레이어가 묻게 만든다.

> 내가 잘못 본 건가?

---

## 2. 게임 핵심 톤

### 장르

- 회사 배경 TUI 텍스트 어드벤처
- 블랙코미디
- 미스터리
- 은근한 코스믹호러
- 회사 괴담 시뮬레이터
- ARG 느낌이 살짝 섞인 사내망 호러

### 감정 목표

직접적인 공포보다 다음 감정을 우선한다.

- 찝찝함
- 의심
- 현실 오류
- 혼란
- 불편한 농담
- 시스템에 대한 불신
- “이거 게임 맞나?”

### 전체 흐름

초반에는 웃기고 이상하다.  
회사 시스템, 공지, 메신저, 로그가 블랙코미디처럼 보인다.

그러나 시간이 지나면서 현실이 미세하게 틀어진다.  
화면, 선택지, 로그, 사내망 메시지가 조금씩 이상해진다.

그리고 다시 아무 일도 없던 것처럼 정상으로 돌아온다.

이 반복이 핵심이다.

```text
초반: 이상한 회사 괴담
중반: 사내망과 공간 오류
후반: 격리된 차원 / 현실 붕괴
```

## 3. 세계관 큰 줄기

### 기본 사건

- 갑작스러운 암전이 발생한다.
- 회사 사람들이 사라진다.
- 회사 건물 내부 공간이 뒤틀린다.
- 사내망은 완전히 죽지 않고 부분적으로 살아 있다.
- 플레이어는 사내 시스템, 로그, 문서, 서버 기록을 통해 상황을 파악한다.
- 후반부에 이것이 단순 정전이나 사고가 아니라 **공간/차원 격리**였음이 드러난다.

### 시작 장소

```text
7층 개발팀 사무실
```

### 초반 탐색 구역

- 내 자리
- 탕비실
- 회의실
- 복도
- 화장실
- 엘리베이터
- 비상계단
- 서버실 입구

### 첫 목표

> 서버실에 들어가라.

서버실은 초반 목표이자, 사내망과 괴이 현상의 핵심 단서를 여는 장소다.

## 4. 게임처럼 보이지 않게 만들기

일반적인 게임 UI는 최소화한다.

피해야 할 것:

- 노골적인 HP 바
- 레벨
- 경험치
- 퀘스트 마커
- 화려한 메뉴
- 몬스터 도감
- 게임적인 보상 UI

대신 사용할 것:

- 터미널
- 로그
- 공지
- 문서
- 메일
- 사내 메신저
- 서버 접속 화면
- 파일 목록
- 시스템 메시지
- 백업 로그
- 에러 메시지

게임 시작도 실제 유틸리티 실행처럼 보이게 한다.

```bash
$ ./escape
```

예시:

```text
[NOTICE]
본 프로그램은 사내 격리 환경 대응용 유틸리티입니다.
```

## 5. 화면 구조

기본 화면은 세 영역으로 나눈다.

```text
┌──────────────────────────────────────────────┐
│ 상단: ASCII 비주얼 / 애니메이션 / 글리치      │
├──────────────────────────────────────────────┤
│ 중단: 스트리밍 상황 묘사 / 로그 / 시스템 메시지│
├──────────────────────────────────────────────┤
│ 하단: 선택지 / 입력창 / 상태 요약             │
└──────────────────────────────────────────────┘
```

### 상단 영역

역할:

- 공간의 모습 제시
- ASCII 비주얼
- 미세한 프레임 변화
- 글리치
- 이상 현상 삽입

예시:

```text
┌──────────────────────────────────────────────┐
│ [7F] 엘리베이터 앞                           │
├──────────────────────────────────────────────┤
│                                              │
│                  ┌────────┐                  │
│                  │  ▒▒▒▒  │                  │
│                  │  ▒7F▒  │                  │
│                  │  ▒▒▒▒  │                  │
│                  │  ┌──┐  │                  │
│                  │  │  │  │                  │
│                  │  │╲ │  │                  │
│                  │  │ ╲│  │                  │
│                  │  └──┘  │                  │
│                                              │
├──────────────────────────────────────────────┤
│ 엘리베이터가 도착했다.                       │
│ 문이 완전히 열리지 않는다.                   │
│ 안쪽에서 축축한 종이 냄새가 난다.            │
│                                              │
│ [시스템] 탑승 권한 확인됨                    │
├──────────────────────────────────────────────┤
│ [1] 엘리베이터에 탄다                        │
│ [2] 문 사이를 자세히 본다                    │
│ [3] 비상계단으로 간다                        │
│ [4] 사내망에 현재 위치를 기록한다            │
└──────────────────────────────────────────────┘
```

### 중단 영역

역할:

- 상황 설명
- 스트리밍 텍스트
- 로그 출력
- 시스템 메시지
- 공지문
- 사내 문서 일부
- 생존자 기록

텍스트는 한 번에 뿌리지 않는다.  
한 줄씩, 혹은 짧은 지연을 두고 출력한다.

예시:

```text
엘리베이터가 도착했다.

문이 열린다.

...

안에서 누군가 숨을 참고 있다.
```

### 하단 영역

역할:

- 선택지
- 입력창
- 현재 상태 요약
- 플레이어 행동 유도

선택지는 단순한 메뉴가 아니라 공포 연출의 일부가 되어야 한다.

핵심 문장:

> 선택지를 그냥 글자로 두지 않고, 누르기 무섭게 만든다.

## 6. 공포 연출 철학

### 기본 원칙

- 대놓고 괴물은 지양한다.
- 큰 점프스케어보다 작은 이상함을 중시한다.
- 화면은 천천히 변한다.
- 이상 현상은 다시 정상으로 돌아온다.
- 플레이어가 스스로 의심하게 만든다.
- UI는 중립적이지 않다.
- 시스템 메시지도 스토리의 일부다.
- 선택지도 세계와 분리되지 않는다.
- 플레이어 입력 자체도 감염될 수 있다.

### 핵심 감각

공포의 중심은 괴물이 아니라 **시스템의 불신**이다.

플레이어는 보통 텍스트와 UI를 믿는다.  
이 게임은 그 믿음을 깨뜨린다.

예시:

```text
[1] 엘리베이터에 탄다
[2] 비상계단으로 간다
[3] 돌아간다
```

플레이어가 `2`를 입력했는데:

```text
입력 확인:
> 엘리베이터로 이동한다
```

이런 순간이 이 게임의 핵심 공포가 된다.

## 7. 주요 연출 기법

### 7.1 미세한 ASCII 프레임 변화

ASCII 아트는 대놓고 변하지 않는다.  
플레이어가 확신하지 못할 정도로만 바뀐다.

```text
│  │  │
```

에서:

```text
│╱ │  │
```

그리고 다시 정상으로 돌아온다.

핵심은 이것이다.

> “방금 바뀐 것 같은데?”가 “괴물이 나타났다”보다 더 강하다.

### 7.2 프레임 애니메이션

장면마다 여러 개의 ASCII 프레임을 둘 수 있다.

```text
assets/ascii/elevator_normal.txt
assets/ascii/elevator_shadow_1.txt
assets/ascii/elevator_shadow_2.txt
assets/ascii/elevator_hand.txt
```

가능한 연출:

- 엘리베이터 문이 천천히 열림
- 형광등이 깜빡임
- 복도 끝 그림자가 한 칸씩 가까워짐
- 화면 구석의 점이 사람 형태로 변함
- 잠깐만 이상한 프레임이 삽입됨

### 7.3 스트리밍 텍스트

한 글자씩, 한 문장씩, 혹은 지연을 두고 출력한다.

중요한 장면에서는 `...` 같은 침묵도 적극 활용한다.

```text
엘리베이터 안쪽은 분명히 비어 있었다.

방금까지는.

문 사이에 걸린 것은 그림자처럼 보였지만,
그림자는 손가락 마디를 가지고 있지 않다.
```

### 7.4 글리치 효과

화면 일부를 깨진 문자로 바꾸거나 시스템 메시지를 오염시킨다.

```text
현재 위치: 7층 복도
현재 시간: 23:41
현재 인원: 1명
현재 인원: 2명
현재 인원: 1명
현재 인원: 2명
```

또는:

```text
[시스템] 엘리베이터 상태: 정상
[시스템] 엘리베이터 상태: 정ㅅ
[시스템] 엘리베이터 상태: 정사아아아아아
[시스템] 엘리베이터 상태: 탑승 권장
```

### 7.5 선택지 오염

선택지가 시간이 지나며 바뀐다.

```text
[1] 엘리베이터에 탄다
```

잠시 후:

```text
[1] 이미 탔다
```

또는:

```text
[1] 엘리베이터에 탄다
[2] 비상계단으로 간다
[3] 돌아간다
[4] 엘리베이터가 당신을 탄다
```

### 7.6 입력 강제

플레이어가 입력한 값과 실제 처리되는 값이 어긋난다.

```text
> 2
```

출력:

```text
잘못된 입력입니다.
> 1
```

단, 이 연출은 남용하면 불쾌하거나 불공정하게 느껴질 수 있으므로 중요한 장면에서만 사용한다.

### 7.7 로그 기반 공포

미래 시점 로그, 자기 이름 로그, 사망 후 로그, 존재하지 않는 사용자 로그 등을 활용한다.

```text
/var/log/survivors/
```

파일 목록:

```text
park_2026-05-15.log
kim_2026-05-15.log
lee_2026-05-15.log
you_2026-05-15.log
you_2026-05-16.log
```

플레이어는 아직 5월 15일에 있는데, 자기 이름의 5월 16일 로그가 있다.

```text
[2026-05-16 03:12:44]
나는 엘리베이터를 타지 말았어야 했다.

[2026-05-16 03:12:45]
그런데 이미 탔다.
```

### 7.8 시스템 메시지 공포

시스템 메시지는 단순 안내가 아니라 세계관 일부다.

```text
[NETWORK QUALITY: GOOD]
```

이후:

```text
[NETWORK QUALITY: POOR]
```

이후:

```text
[CONNECTION TO REALITY UNSTABLE]
```

대표 톤:

```text
[시설팀 공지]
13층에서 자기 자신을 목격하신 경우,
눈을 마주치지 마시기 바랍니다.
```

## 8. 사내망 시스템

사내망은 이 게임의 핵심 탐색 도구이자 공포 장치다.

포함 가능한 요소:

- 사내 메신저
- 공지
- 문서
- 서버 로그
- 생존자 로그
- 시설팀 메시지
- 근태 시스템
- 백업 파일
- Git commit message
- branch 이름
- README
- workspace 디렉토리
- cron job
- ping 결과
- mount 실패
- 서버 접속 오류

핵심 감각:

> 사람은 사라졌는데 시스템은 정상 작동한다.

예시:

```text
[근태 시스템]
현재 상태: 근무 중
퇴근 승인: 대기 중
```

이런 장면은 웃기지만 기분 나쁘다.  
이 게임의 블랙코미디와 호러가 동시에 살아나는 지점이다.

## 9. 현실감 연출

### 실제 회사 환경을 읽는 느낌

이 게임의 강력한 무기는 현실과 허구의 경계 흐림이다.

플레이어가 다음과 같이 느끼게 만드는 것이 목표다.

> 이게 게임 안 정보인지, 내 실제 환경에서 나온 건지 모르겠다.

사용할 수 있는 가짜 현실 요소:

- `feature/fix-elevator-bug`
- `임시 패치. 원인은 아직 모름.`
- `회의록_최종_v2_진짜최종.docx`
- `조직개편_공유용.pptx`
- `README.txt`
- `workspace_final_final`
- `fix_final_REAL_final.cpp`
- `workspace_use_this_one`

예시:

```text
[workspace-sync]
최근 수정된 문서:
- 회의록_최종_v2_진짜최종.docx
- 조직개편_공유용.pptx
- README.txt

README의 마지막 문장이 자꾸 떠오른다.
"문이 닫히기 전에 나가야 한다."
```

### 안전 원칙

> 실제 접근보다 접근한 척이 더 무섭다.

실제 민감정보를 읽거나 접근하는 것은 피해야 한다.  
대신 회사 환경에서 흔히 볼 법한 더미 데이터를 사용한다.

목표는 “설마?”라는 감각이지, 실제 개인정보 침범이 아니다.

원칙:

- 실제 문서, 실제 파일 내용, 실제 사내 자료를 읽지 않는다.
- 공개 repo에는 현실 최종 위치, 민감한 회사 정보, 개인 식별 정보를 넣지 않는다.
- 사용자명, branch, workspace 같은 요소는 가짜 샘플 또는 안전하게 제한된 값만 사용한다.
- 민감정보처럼 보이는 연출은 더미 데이터로 처리한다.

## 10. 현실 기반 랜덤 요소

실행 시간, 요일, 환경명처럼 현실과 연결된 듯한 요소를 사용할 수 있다.

가능한 연출:

- 새벽에 실행하면 다른 문구 출력
- 금요일 밤이면 야근자 로그 증가
- 특정 시간대에 사내망 응답이 이상해짐
- 장시간 아무 입력이 없으면 화면이 변함

예시:

```text
밖이 조용하다.
원래 이 시간 회사는 조금 더 시끄러웠던 것 같은데.
```

주의:

- 실제 시간 기반 이벤트는 보조 연출로만 사용한다.
- 핵심 스토리는 고정 흐름으로 진행한다.
- 테스트를 위해 seed 또는 debug time override를 둔다.

## 11. 장면 설계 방식

장면은 코드가 아니라 데이터 중심으로 관리하는 것이 좋다.

```json
{
  "id": "elevator_7f_first",
  "title": "7F 엘리베이터 앞",
  "visual_frames": [
    "elevator_normal",
    "elevator_shadow_1",
    "elevator_shadow_2"
  ],
  "text": [
    "엘리베이터가 도착했다.",
    "문이 완전히 열리지 않는다.",
    "안쪽에서 축축한 종이 냄새가 난다."
  ],
  "choices": [
    {
      "label": "엘리베이터에 탄다",
      "next": "elevator_inside"
    },
    {
      "label": "문 사이를 자세히 본다",
      "next": "elevator_peek"
    },
    {
      "label": "비상계단으로 간다",
      "next": "stairwell_7f"
    }
  ]
}
```

추천 콘텐츠 단위:

```text
/content/encounters/
```

예시:

```text
elevator_7f.md
printer_room.md
meeting_room_loop.md
stairwell_7f.md
server_room_entrance.md
```

각 encounter는 다음 정보를 가진다.

- 장소
- 비주얼 프레임
- 텍스트
- 선택지
- 글리치 조건
- 시간 경과 이벤트
- 연결되는 다음 장면
- 사내망/로그 단서

- 형광등 깜빡임

- 사내 방송 로그

- 현재 인원 수 오염

- “이미 탔다” 출력

## 16. 충돌 또는 정리 필요한 아이디어

현재 아이디어들은 큰 방향에서는 잘 맞는다.  
다만 실제 제작 전에 아래 기준을 정해야 한다.

### 16.1 현실감 연출 vs 개인정보/보안 불쾌감

아이디어:

- 게임이 실제 회사 환경을 읽는 것처럼 보이게 한다.
- 사용자명, workspace, commit message, README, branch, 공용 문서 등을 활용한다.

충돌 가능성:

- 실제로 민감정보를 읽으면 불쾌감, 보안 문제, 신뢰 문제가 생길 수 있다.

정리 제안:

- 실제 접근은 하지 않는다.
- 더미 데이터, 가짜 파일명, 안전하게 제한된 환경 변수 정도만 사용한다.

원칙:

> 실제로 읽지 말고, 읽은 것처럼 느끼게 한다.

### 16.2 UI가 플레이어를 속이는 것 vs 조작 불공정성

아이디어:

- 선택지 오염, 입력 강제, 잘못된 입력 확인 등을 사용한다.

충돌 가능성:

- 너무 자주 쓰면 플레이어가 짜증을 느낀다.
- “내 선택이 의미 없다”고 느낄 수 있다.

정리 제안:

- 입력 강제는 매우 제한적으로 사용한다.
- 중요한 장면에서만 사용한다.
- 사용 후 세계관적 이유를 암시한다.
- 모든 선택을 무효화하지 않는다.

### 16.3 블랙코미디 vs 코스믹호러

아이디어:

- 처음엔 웃기고 이상하다.
- 나중에는 차원 격리, 현실 오류, 코스믹호러로 확장된다.

충돌 가능성:

- 농담이 너무 강하면 공포가 약해진다.
- 반대로 공포가 너무 강하면 회사 블랙코미디 정체성이 사라진다.

정리 제안:

```text
초반: 블랙코미디 비중 높음
중반: 미스터리와 공간 오류 증가
후반: 코스믹호러와 현실 붕괴 비중 증가
```

### 16.4 대놓고 괴물 금지 vs 손가락/그림자 연출

아이디어:

- 대놓고 괴물을 보여주지 않는다.
- 하지만 엘리베이터 문틈의 손가락, 그림자, 사람 형태 ASCII 등은 사용한다.

충돌 가능성:

- 손가락이나 사람 형상이 너무 명확하면 “대놓고 괴물”처럼 느껴질 수 있다.

정리 제안:

- 괴이는 완전히 보여주지 않는다.
- 부분만 보여준다.

허용되는 표현:

- 문틈의 손가락 같은 것
- 복도 끝의 점
- 사람처럼 보이는 ASCII
- 로그 속 묘사
- 시스템이 잘못 세는 인원 수

피해야 할 표현:

- 괴물 전체 모습
- 노골적인 추격전
- 몬스터 이름과 체력
- 정체를 완전히 설명하는 장면

### 16.5 화면이 계속 변하는 연출 vs 플레이어 피로

아이디어:

- 선택지를 띄운 상태에서도 화면이 가만히 있지 않는다.
- 시간이 지나면 문틈이 벌어지고, 소리가 나고, 프레임이 변한다.

충돌 가능성:

- 플레이어가 천천히 읽고 생각할 시간을 빼앗긴다고 느낄 수 있다.

정리 제안:

- 시간 경과 연출은 압박감이 필요한 장면에서만 사용한다.
- 실제 진행을 강제하기보다는 분위기 변화 중심으로 둔다.

### 16.6 게임 UI 최소화 vs 상태 관리 필요

아이디어:

- HP, 레벨, 퀘스트 UI를 없애고 실제 터미널처럼 만든다.

기존 방향과의 정합성:

- 현재 기획에는 체력, 정신력, 배터리, 허기, 갈증 같은 생존 자원이 존재한다.
- 따라서 자원 자체를 없애기보다, 표시 방식을 게임 HUD처럼 보이지 않게 해야 한다.

정리 제안:

- 상태값은 사내 시스템 로그, 진단값, 단말기 배터리, 컨디션 리포트처럼 보여준다.

예시:

```text
[LOCAL STATUS]
신체 반응: 둔화
집중도: 불안정
단말기 전원: 43%
최근 식수 기록: 없음
```

또는:

```text
[격리 대응 유틸리티]
현재 위치: 7F 개발팀 사무실
사내망: 제한적 연결
권장 행동: 서버실 접근 권한 확인
```

### 16.7 랜덤/현실 시간 이벤트 vs 테스트 가능성

아이디어:

- 실제 시간이나 요일에 따라 이벤트가 달라진다.

충돌 가능성:

- 테스트와 디버깅이 어려워진다.
- 플레이어마다 중요한 장면을 놓칠 수 있다.

정리 제안:

- 실제 시간 기반 이벤트는 보조 연출로만 사용한다.
- 핵심 스토리는 고정 흐름으로 진행한다.
- 개발용으로 seed 또는 debug time override를 둔다.

### 16.8 스토리보다 연출 중심 vs 엔딩/세계관 필요

아이디어:

- 차별점은 스토리보다 “터미널 자체가 무섭다”는 감각이다.

충돌 가능성:

- 연출만 있고 중심 서사가 약하면 후반 몰입이 떨어질 수 있다.

정리 제안:

스토리는 복잡할 필요는 없지만, 플레이어가 따라갈 중심축은 있어야 한다.

```text
암전 발생
→ 사람 사라짐
→ 사내망 생존
→ 서버실 접근
→ 로그에서 격리 현상 확인
→ 공간/차원 오류 확인
→ 탈출 또는 잔류 선택
```

## 17. 정리된 최종 방향성

이 게임은 다음 방향으로 잡는 것이 가장 좋다.

> 회사 괴담처럼 시작해서, 사내망 로그와 TUI 인터페이스를 통해 현실이 틀어지고 있음을 보여주는 블랙코미디 미스터리 호러.

가장 중요한 차별점은 이것이다.

> 무서운 장소를 보여주는 게임이 아니라, 믿고 있던 터미널과 회사 시스템이 플레이어를 속이는 게임.

제작 기준:

- 터미널은 단순 UI가 아니라 세계관의 일부다.
- 로그는 설명문이 아니라 공포 장치다.
- 선택지는 행동 메뉴가 아니라 불안 요소다.
- 사내망은 정보 제공 장치이자 괴이 현상의 통로다.
- 실제 회사 데이터를 읽는 느낌은 주되, 실제 민감정보는 건드리지 않는다.
- 괴물보다 시스템 오류, 현실 오류, 회사 시스템의 무표정함이 더 무섭다.
- 플레이어가 계속 “내가 잘못 본 건가?”라고 느끼게 만든다.
