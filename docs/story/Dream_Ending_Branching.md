# 꿈 엔딩 분기 설계

## 상태

이 문서는 `idea_box/done/2026-05-22-dream-ending-branching.md`를 story/design 후보로 승격한 문서다.
현재 런타임 엔딩 YAML이나 Rust/Python 엔딩 판정에 `dream`/`epilogue` 타입을 추가했다는 뜻은 아니다.

현재 판정:

- 채택 범위: 확정 구현이 아니라 후속 엔딩/스토리팩 작성용 설계 계약 후보
- 런타임 변경: 없음
- 구현 전제: 구현한다면 Rust GameCore와 공개 YAML/content bundle을 source of truth로 둔다
- 공개 안전: 실제 사용자 메모, 실제 회사/직원/위치, private 현실 단서는 넣지 않는다

## 핵심 원칙

꿈 엔딩은 “다 꿈이었다”로 플레이 경험을 무효화하는 반전이 아니다.
꿈은 플레이어가 현실에서 외면하던 문제를 상징적으로 체험하게 하는 해석 장치다.

핵심 문장:

```text
사건은 꿈이어도 된다.
감정은 진짜여야 한다.
선택은 남아야 한다.
깨어난 뒤의 주인공은 이전과 달라야 한다.
```

따라서 꿈 엔딩은 다음 구조를 기본형으로 둔다.

```text
꿈속 최종 선택
→ 상징적 붕괴 또는 끔찍한 결과
→ 깨어남
→ 안도
→ 현실에서 같은 문제가 작게 재등장
→ 마지막 선택 또는 자동 판정
→ 짧은 후일담 컷
→ 엔딩명 표시
```

## 현재 엔딩 시스템과의 관계

현재 공개 런타임 엔딩은 `docs/content/Ending_List.md` 기준으로 다음 계열을 가진다.

- `failure`
- `escape`
- `hidden`
- `truth`
- `conquest`

꿈 엔딩은 이 목록을 즉시 대체하지 않는다.
후속 구현 시 선택지는 세 가지다.

1. 기존 엔딩 위에 후일담 레이어로 붙인다.
   - 예: `truth` 엔딩 후 “깨어난 뒤 현실에서 기록을 보낼지”를 짧게 정산한다.
2. 특정 스토리팩 전용 엔딩 문법으로 둔다.
   - 예: 반복 업무일/문서오염/회의실팩에서만 꿈-현실 재시험 구조를 사용한다.
3. 새 엔딩 타입 후보를 만든다.
   - 예: `dream`, `epilogue`, `loop_failure`
   - 이 경우 `docs/dev/Data_Schema.md`, YAML, Rust `EndingDef`, Web/terminal renderer 표시 계약을 별도 slice로 갱신해야 한다.

현재 문서 승격 단계에서는 1 또는 2를 우선 검토하고, 새 타입 추가는 별도 구현 결정이 있을 때만 연다.

## 작성용 판단 축

다음 값들은 아직 런타임 변수로 확정하지 않는다.
콘텐츠 작성과 엔딩 후보 분류를 위한 판단 축이다.

```text
helped_colleague_count
abandoned_colleague_count
accepted_responsibility
shifted_blame
faced_truth
deleted_evidence
sent_help_message
refused_unfair_order
chose_self_preservation
chose_escape
chose_return_to_work
chose_sleep_again
recognized_dream_logic
entered_b1_storage
final_input_word
```

`final_input_word`는 꿈속 마지막 문이나 터미널에 입력한 상징어다.
구현 전에 자유 입력을 받을지, 선택지 id를 상징어로 매핑할지 결정해야 한다.
기본 구현은 선택지 기반 매핑이 안전하다.

예시 매핑:

| 꿈속 상징 | 현실 행동 후보 |
|---|---|
| `책임` | 부당한 책임을 거절한다. |
| `분노` | 폭발 대신 기록/방향 설정으로 바꾼다. |
| `회사` | 병가, 면담, 사직서 초안, 이직 준비를 검토한다. |
| `나` | 자기비난을 멈추고 자기보호를 선택한다. |
| `동료` | 동료에게 먼저 연락하거나 함께 들어간다. |
| `사과` | 외면했던 인물에게 첫 문장을 보낸다. |
| `침묵` | 회피/불안/루프 배드엔딩으로 처리한다. |

## 엔딩 후보군

원문 아이디어의 15개 후보는 다음 감정군으로 보존한다.

| 후보 | 감정/기능 | 구현 우선도 메모 |
|---|---|---|
| 아직 출근 전 | 두 번째 기회, 관계 회복 가능 | 동료 루트와 결합하기 좋다. |
| 읽음만 남았다 | 회피 지속, 배드엔딩 | 메시지/메신저 UI와 잘 맞다. |
| 병가 | 자기보호, 조용한 생존 | 플레이어 회복/자기보호 엔딩 후보. |
| 제 일이 아닙니다 | 부당한 책임 거절 | 회사 블랙코미디 톤과 강하게 맞는다. |
| 초안 | 미완의 탈출 | 현실적 여운을 남기는 노멀 엔딩 후보. |
| 발송 | 결단, 기록/제보 | truth 또는 real escape aftermath와 결합 가능. |
| 다시 잠 | 회피 루프 | loop failure 후보. |
| 같이 들어가자 | 관계 회복 | 동료 구조/동행 루트와 결합 가능. |
| 증거는 남아 있다 | 진실 직면 | 증거/기록 루트와 결합 가능. |
| 자동응답 | 자기 상실 | 회사 시스템 동화 배드엔딩 후보. |
| 미처리 감정 보관소 | 내면 직면 | `final_input_word` 실험 후보. |
| 오늘도 같은 꿈 | 완전 회피 | 반복 업무일/루프 재난과 결합 가능. |
| 처음으로 도움을 요청했다 | 자기보호, 희망 | 직접 조언처럼 보이지 않게 서사 문맥 유지. |
| 문밖의 아침 | 조용한 해방 | 굿엔딩 후보. |
| 꿈이라서 다행이었다 | 종합형 메인 굿엔딩 후보 | 여러 후일담을 묶는 상위 컨셉으로 적합. |

## 구현 후보 slice

꿈 엔딩을 실제 구현으로 승격할 때는 한 번에 전체 15개를 넣지 않는다.
권장 첫 slice:

1. `docs/content/Ending_List.md`에 “꿈 엔딩은 아직 런타임 미구현” 상태를 명시하거나 별도 후보 섹션을 둔다.
2. 공개 YAML에 기존 엔딩 하나의 후일담 텍스트 후보만 추가할지, 별도 `ending_epilogues.yaml`을 만들지 결정한다.
3. Rust GameCore에 새 타입을 추가하지 않고도 가능한지 먼저 확인한다.
4. Web Storybook/terminal이 같은 `ScenePage.mode: ending`으로 후일담을 표시할 수 있는지 검증한다.
5. 테스트는 꿈속 선택 → ending page → 후일담 page의 최소 scripted smoke 하나로 시작한다.

새 `kind`를 추가해야 한다면 그때 별도 schema/code PR로 분리한다.

## 품질 기준

- “꿈이었다”가 아니라 “다르게 읽힌다”는 감각을 준다.
- 꿈속 선택이 현실 행동으로 번역된다.
- 깨어난 뒤 주인공이 이전과 다르게 행동하거나, 바뀌지 않는다면 그것이 명확한 배드엔딩으로 작동한다.
- 후일담은 장황한 설명보다 메시지, 손동작, 커서, 알림, 출근길 같은 구체물로 보여준다.
- 배드엔딩도 후일담을 가진다.
- 현실 행동은 게임 속 캐릭터의 서사 선택으로만 다루며, 실제 사용자에게 병가/퇴사/제보 같은 행동을 직접 지시하는 문맥으로 쓰지 않는다.

## 공개 안전 경계

꿈 엔딩이 현실 연결 히든 엔딩과 결합될 수 있어도 다음은 공개 문서/런타임 데이터에 쓰지 않는다.

- 실제 사용자 사적 메모
- 실제 회사명, 직원명, 조직명
- 실제 위치, 층수, 좌석, 보안 구조
- 실제 내부망/계정/시스템 정보
- private/local 현실 단서 최종 문장

현실 연결과 섞이는 경우 `docs/story/Reality_Link.md`의 공개/비공개 경계를 따른다.
