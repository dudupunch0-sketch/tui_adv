# 현실 탈출 엔딩 분기 설계

## 상태

이 문서는 `idea_box/inbox/2026-05-22-real-escape-ending-branching.md`를 story/design 후보로 승격한 문서다.
현재 런타임 엔딩 YAML이나 Rust/Python 엔딩 판정에 `real_escape`/`post_escape`/`aftermath` 타입을 추가했다는 뜻은 아니다.

현재 판정:

- 채택 범위: 후속 엔딩 확장용 설계 계약 후보
- 런타임 변경: 없음
- 구현 전제: 구현한다면 Rust GameCore와 공개 YAML/content bundle을 source of truth로 둔다
- 공개 안전: 실제 회사, 실제 직원, 실제 위치, 실제 내부 시스템, private 현실 단서는 넣지 않는다

## 한 줄 정의

현실 탈출 엔딩에서 탈출은 최종 보상이 아니라 결과 정산의 시작 조건이다.

핵심 문장:

```text
탈출은 문을 나서는 순간 끝나는 것이 아니라,
문밖으로 무엇을 가져왔는지로 완성된다.
```

좋은 현실 탈출 엔딩은 다음 질문을 남긴다.

```text
밖으로 나왔다.
그런데 누구를 데리고 나왔는가?
무엇을 증명할 수 있는가?
무엇을 버리고 나왔는가?
무엇이 아직 안에 남아 있는가?
```

## 꿈 엔딩과의 대비

| 구분 | 꿈 엔딩 | 현실 탈출 엔딩 |
|---|---|---|
| 사건 성격 | 악몽, 내면의 시뮬레이션, 회귀성 경험 | 실제로 일어난 감금/이상현상/거래/실종/폭로 |
| 핵심 감정 | 안도, 두 번째 기회, 재선택 | 생존, 대가, 책임, 후폭풍 |
| 엔딩 이후 | 깨어난 뒤 현실 선택이 중요 | 나온 뒤 현실의 결과 정산이 중요 |
| 플레이어 감정 | “꿈이라서 다행이다. 이제 다르게 선택할 수 있다.” | “나왔다. 하지만 이제 진짜 문제가 시작된다.” |

두 문법은 서로 배타적이지 않다.
같은 스토리팩 안에서 “꿈으로 끝나는 분기”와 “현실 탈출로 끝나는 분기”가 대비될 수 있다.

## 현재 엔딩 시스템과의 관계

현재 공개 런타임 엔딩은 `docs/content/Ending_List.md` 기준으로 `failure`, `escape`, `hidden`, `truth`, `conquest` 계열이다.
현실 탈출 엔딩 분기는 이 목록을 즉시 대체하지 않는다.

후속 구현 후보:

1. 기존 `escape` 엔딩에 후일담 결과 정산을 붙인다.
2. 기존 `truth` 엔딩과 결합해 증거/공개/감사 개시 분기를 만든다.
3. `hidden` 현실 연결 엔딩과 결합하되 private-only 단서는 공개 데이터로 내보내지 않는다.
4. 탈출 직전 마지막 선택과 탈출 후 후일담을 분리한 2단 구조를 둔다.
5. 필요할 때만 `real_escape`, `post_escape`, `aftermath` 같은 새 타입을 별도 schema/code slice로 검토한다.

현재 문서 승격 단계에서는 “후일담 결과 정산 설계”까지만 확정한다.
새 엔딩 타입과 새 런타임 필드는 아직 확정하지 않는다.

## 작성용 판단 축

다음 값들은 아직 런타임 변수로 확정하지 않는다.
콘텐츠 작성과 엔딩 후보 분류를 위한 판단 축이다.

```text
escaped
alive
rescued_count
rescued_key_npc
abandoned_key_npc
witness_count
has_evidence
evidence_level
evidence_uploaded
evidence_destroyed
evidence_traded
truth_action
company_response
self_sacrifice
identity_preserved
employee_record_exists
company_assimilation
post_escape_risk
next_play_unlock
```

`evidence_level` 후보:

| 값 | 의미 |
|---:|---|
| 0 | 없음 |
| 1 | 정황 증거 |
| 2 | 내부 자료, 녹취, 로그 등 실질 증거 |
| 3 | 원본 로그, 핵심 파일, 다중 증언, 외부 전송 완료 |

`truth_action` 후보:

```text
none | silent | private_share | public_leak | official_report | company_trade | destroy
```

## 엔딩 선택 우선순위 후보

여러 조건이 동시에 충족될 수 있으므로 후속 구현 전 우선순위를 설계해야 한다.

1. 자기희생/잔류 계열은 일반 탈출보다 우선한다.
2. 회사 동화 또는 증거 거래는 단순 성공으로 보이지 않게 거래/동화 계열을 우선한다.
3. 결정적 증거, 핵심 증인, 공식 신고/공개가 함께 있으면 트루엔딩 계열로 둔다.
4. 증거가 있지만 혼자 탈출했다면 증거 엔딩 계열로 둔다.
5. 동료를 구했지만 증거가 부족하면 증인/동료 구출 엔딩 계열로 둔다.
6. 탈출했지만 증거도 동료도 없으면 생존/불신 엔딩 계열로 둔다.
7. 탈출했지만 정체성이 보존되지 않았으면 정체성 상실 계열을 섞는다.
8. 탈출하지 못했으면 잔류/실종/희생/포획 계열로 둔다.

우선순위는 절대 규칙이 아니라 충돌 방지용 가이드다.
특정 스토리팩의 핵심 테마가 있으면 그 테마를 우선한다.

## 후일담 형식

| 형식 | 잘 맞는 엔딩 |
|---|---|
| 뉴스 기사 | 공개 폭로, 감사, 사건 확대 |
| 회사 공지 | 은폐, 부인, 법적 대응, 사내 통제 |
| 메신저 로그 | 동료 구출, 죄책감, 관계 회복 |
| 경찰/감사 진술서 | 현실성, 공식성, 불신 |
| 시스템 리포트 | TUI 스타일, 결과 정산, 상태 변수 표시 |
| 사원 시스템 로그 | 정체성 상실, 회사 동화, 추적 |

예시 형식:

```text
[POST-ESCAPE REPORT]

survivor_count: 2
evidence_uploaded: true
internal_audit: initiated
company_response: denial
employee_status: terminated
risk_level: ongoing

ENDING: 감사 개시
```

이 예시는 공개-safe placeholder다.
실제 회사명, 실제 기관명, 실제 직원/위치는 쓰지 않는다.

## 대표 후보군

| 후보 | 핵심 감정 | 구현 우선도 메모 |
|---|---|---|
| 정문 밖 | 살았지만 아무도 믿어주지 않음 | 기본 escape 후일담 후보. |
| 업로드 완료 | 이겼지만 이제 쫓김 | 증거/사내망 루트와 결합 가능. |
| 두 명 이상 퇴근 가능 | 회사는 무너지지 않았지만 증인이 생김 | 동료/NPC 시스템과 결합 가능. |
| 감사 개시 | 제대로 해냈지만 싸움은 시작됨 | truth 루트 굿엔딩 후보. |
| 임원 엘리베이터 | 탈출이 아니라 승격됨 | conquest/회사 동화 배드엔딩 후보. |
| 없었던 야근 | 안전하지만 침묵의 대가가 남음 | 회사 공지/은폐 후일담 후보. |
| 서버실에 남은 사람 | 나는 못 나갔지만 문은 열었음 | 자기희생/잔류 후보. |
| 출입기록 없음 | 나왔지만 내가 있었다는 증거가 없음 | 정체성 상실 후보. |
| 회수팀 호출 | 탈출했지만 퇴근은 승인되지 않음 | 추적/후속 위험 후보. |
| 증인 보호 | 침묵이 완벽하지 않게 됨 | 동료 증언 후보. |
| 퇴사 처리 통지 | 나가는 방식도 회사가 정함 | 보복/기록 조작 후보. |
| 재입사 제안 | 회사가 아직 나를 후보자로 봄 | 회유/거래 후보. |
| 서버와 함께 묻다 | 회사를 상처 입혔지만 진실도 죽임 | destroy 계열 후보. |
| 마지막 퇴근자 | 혼자 살아남은 죄책감 | 동료 포기 대가 후보. |
| 문밖으로 가져간 것 | 끝나지 않았지만 빈손은 아님 | 대표 노멀/굿엔딩 후보. |

## 구현 후보 slice

현실 탈출 엔딩을 실제 구현으로 승격할 때는 한 번에 전체 후보를 넣지 않는다.
권장 첫 slice:

1. 현재 `escape_commute`, `escape_rooftop_signal`, `escape_parking_lot`, `escape_lobby_revolving_door` 중 하나만 고른다.
2. 그 엔딩의 후일담 후보를 공개-safe 텍스트로 작성한다.
3. 새 상태 필드를 만들기 전에 기존 flags/items/clues로 충분히 표현 가능한지 확인한다.
4. Rust GameCore의 기존 `ScenePage.mode: ending`으로 표시 가능한지 먼저 검증한다.
5. 새 `kind` 또는 `aftermath` field가 꼭 필요하면 `docs/dev/Data_Schema.md`, YAML, Rust loader, Web/terminal rendering을 별도 slice로 갱신한다.

## 품질 기준

- 탈출 후 모든 문제가 즉시 해결되지 않는다.
- 증거가 없는데 모두가 믿어주는 식으로 선택의 가치를 약화하지 않는다.
- 동료를 버렸다면 감정적/서사적 대가가 남는다.
- 거래 엔딩은 단순 성공이 아니라 불쾌한 보상이어야 한다.
- 다음 플레이 영향은 보너스/흔적이어야 하며 현재 엔딩의 감정적 완결성을 해치지 않는다.
- 플레이어가 “나왔다. 하지만 이제부터가 진짜다”라고 느낄 수 있어야 한다.

## 공개 안전 경계

현실 탈출 엔딩은 현실 연결 히든 엔딩과 결합하기 쉽다.
따라서 다음은 공개 문서/런타임 데이터에 쓰지 않는다.

- 실제 사용자 사적 메모
- 실제 회사명, 직원명, 조직명
- 실제 위치, 층수, 좌석, 보안 구조
- 실제 내부망/계정/시스템 정보
- private/local 현실 단서 최종 문장

현실 연결과 섞이는 경우 `docs/story/Reality_Link.md`의 공개/비공개 경계를 따른다.
