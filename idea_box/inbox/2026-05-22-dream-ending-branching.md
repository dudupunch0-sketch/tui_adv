# 꿈 엔딩 기반 엔딩 분기 후보

## 원문/요지

사용자는 `idea_box/dream_ending_branch_ideas.md`에 “꿈 엔딩”을 활용한 엔딩 분기 컨셉을 추가했다. 핵심은 흔히 나쁜 평가를 받는 “다 꿈이었다” 반전을 그대로 쓰는 것이 아니라, 꿈속의 사건을 주인공이 현실에서 외면하던 문제를 상징적으로 체험하는 구조로 사용하는 것이다.

원문이 강조한 핵심 문장:

```text
사건은 꿈이어도 된다.
감정은 진짜여야 한다.
선택은 남아야 한다.
깨어난 뒤의 주인공은 이전과 달라야 한다.
```

이 문서는 해당 원문을 나중의 설계/개발 agent가 바로 검토할 수 있도록 정리한 아이디어 후보 문서다. 이 아이디어는 아직 확정 요구사항이 아니며, 기존 런타임 엔딩 목록을 즉시 대체하라는 뜻도 아니다.

## 핵심

꿈 엔딩은 사건을 무효화하는 반전이 아니라, 플레이어의 선택을 현실의 마지막 행동으로 다시 검산하는 장치로 쓴다.

권장 구조:

```text
1. 꿈속 최종 선택
2. 끔찍한 결과 또는 상징적 붕괴
3. 깨어남
4. 안도
5. 현실에서 같은 문제 재등장
6. 플레이어의 마지막 선택
7. 짧은 후일담 컷
8. 엔딩명 표시
```

중요한 것은 5~7번이다. “꿈에서 깼다”로 끝내면 플레이 경험이 무효화된다. 깨어난 뒤 현실에서 같은 문제가 작게 다시 등장하고, 플레이어가 마지막으로 한 번 더 선택해야 한다.

## 기대 효과

- 기존 회사-아포칼립스/오피스 호러 톤에 “꿈의 문법”을 추가해 초현실성을 강화할 수 있다.
- 플레이어가 “내 선택이 무효화됐다”가 아니라 “꿈속 선택이 현실 행동으로 번역됐다”고 느끼게 만들 수 있다.
- 단순 탈출/정복/진실 엔딩 외에 관계 회복, 자기보호, 책임 거절, 기록/제보, 회피 루프 같은 감정적 엔딩 축을 만들 수 있다.
- TUI/fake-terminal 연출과 잘 맞는다. 터미널 입력어, 시스템 메시지, 미처리 업무 카운터, 자동응답 계정 같은 요소가 꿈과 현실을 연결할 수 있다.
- 엔딩 후일담을 짧은 행동 컷으로 만들 수 있어, 긴 설명 없이도 여운을 남길 수 있다.

## 구현 후보

### 1. 설계 문서 후보로 먼저 승격

바로 런타임 데이터에 넣기보다, 먼저 `docs/design/` 또는 `docs/story/` 아래에 “꿈 엔딩 분기 설계” 문서로 승격하는 방식을 검토할 수 있다.

후보 문서명:

```text
docs/story/Dream_Ending_Branching.md
# 또는
docs/design/Dream_Ending_Branching.md
```

그 문서에서는 다음을 확정/비확정으로 나눠 정리하면 좋다.

- 꿈 엔딩을 메인 스토리팩의 핵심 구조로 채택할지
- 기존 탈출/정복/진실/히든 현실 연결 엔딩과 병렬 후보로 둘지
- 꿈 엔딩을 특정 스토리팩 전용으로 둘지, 전체 엔딩 시스템의 공통 문법으로 둘지
- 후일담 선택지를 실제 게임 선택지로 구현할지, 엔딩 텍스트 내부 판정으로 처리할지

### 2. 현재 엔딩 데이터와의 연결 후보

현재 공개 문서 기준 런타임 엔딩은 `docs/content/Ending_List.md`에 정리되어 있다. 꿈 엔딩을 구현할 경우, 기존 엔딩을 즉시 삭제/대체하지 말고 다음 중 하나를 검토한다.

- 기존 `escape`, `truth`, `conquest`, `hidden`, `failure` 엔딩 위에 후일담 레이어를 추가한다.
- 꿈 엔딩 전용 타입을 추가한다. 예: `dream`, `epilogue`, `loop_failure`.
- 특정 조건을 만족한 엔딩에서만 “깨어난 뒤 마지막 선택”을 후속 장면으로 붙인다.
- “꿈 엔딩 스토리팩”을 별도 데이터팩으로 두고 기존 기본 엔딩과 분리한다.

주의: 이 아이디어는 아직 기존 `src` 엔딩 판정이나 YAML 스키마를 바꾸라는 지시가 아니다. 구현 전에 설계 결정이 필요하다.

### 3. 상태 변수 후보

원문에서 제안한 추상 변수:

```text
helped_colleague_count
abandoned_colleague_count
accepted_responsibility
shifted_blame
faced_truth
deleted_evidence
sent_help_message
refused_unfair_order
chose_self_preservation
chose_escape
chose_return_to_work
chose_sleep_again
recognized_dream_logic
entered_b1_storage
final_input_word
```

`final_input_word`는 마지막 문이나 터미널에 입력한 단어다. 후일담에서는 이 단어를 현실 행동으로 번역한다.

예시 매핑:

| 꿈속 입력/상징 | 현실 행동 후보 |
|---|---|
| `책임` | 부당한 책임을 거절한다. |
| `분노` | 즉시 폭발하지 않고 기록/방향 설정으로 바꾼다. |
| `회사` | 병가, 면담, 사직서 초안, 이직 준비를 검토한다. |
| `나` | 자기비난을 멈추고 자기보호를 선택한다. |
| `동료` | 동료에게 먼저 연락하거나 같이 들어간다. |
| `사과` | 외면했던 인물에게 첫 문장을 보낸다. |
| `침묵` | 회피/불안/배드엔딩으로 처리한다. |
| 서버 로그 저장 | 증거 보관, 공유, 제보로 이어진다. |
| 자동응답 계정 | 현실에서도 자동 순응하는 배드엔딩이 된다. |
| 다시 잠듦 | 회피 루프가 된다. |

### 4. 엔딩 작성 포맷 후보

나중에 콘텐츠 agent가 엔딩 초안을 만들 때는 각 엔딩을 다음 항목으로 작성하면 좋다.

```text
엔딩명
엔딩 유형
진입 조건
꿈속 최종 장면
깨어나는 장면
현실의 재시험 장면
마지막 선택 또는 자동 판정
후일담
마지막 한 줄
플레이어가 느껴야 할 감정
```

## 엔딩 후보 목록

원문은 15개 엔딩 패턴을 제안한다. 아래 목록은 설계자가 빠르게 훑을 수 있도록 요약한 것이다.

| 번호 | 엔딩명 | 유형/감정 | 핵심 조건 후보 | 현실 후일담 핵심 |
|---:|---|---|---|---|
| 1 | 아직 출근 전 | 두 번째 기회 / 관계 회복 가능 | 꿈속에서 동료를 버렸지만 후회 인식 | `{동료A}`가 “오늘 같이 들어가 줄 수 있어?”라고 묻고, 플레이어가 다시 선택한다. |
| 2 | 읽음만 남았다 | 회피 지속 / 배드엔딩 | 동료를 반복적으로 외면하고 죄책감도 인정하지 않음 | 메시지를 읽고도 답장하지 않는다. |
| 3 | 병가 | 자기보호 / 조용한 생존 | 자신을 과도하게 몰아붙이다가 멈춤을 인식 | 병가 메시지를 보내고 알람을 끈다. |
| 4 | 제 일이 아닙니다 | 부당한 책임 거절 / 성장 | 책임과 죄책감의 차이를 이해 | 상사의 부당한 요구에 “그건 제 일이 아닙니다”라고 답한다. |
| 5 | 초안 | 미완의 탈출 / 현실적 여운 | 탈출을 원하지만 즉각 결단까지는 못 감 | 사직서 또는 변화 문서의 첫 문장을 쓴다. |
| 6 | 발송 | 결단 / 강한 탈출 | 진실을 외면하지 않고 행동 근거를 모음 | 사직서/제보/면담 요청서를 실제로 전송한다. |
| 7 | 다시 잠 | 회피 / 루프 배드엔딩 | 핵심 진실을 외면하고 현실 선택을 피함 | 알람을 끄고 다시 자며 꿈속 사무실로 돌아간다. |
| 8 | 같이 들어가자 | 관계 회복 / 공동 생존 | 동료를 끝까지 도왔고 혼자 탈출하지 않음 | 회사 앞에서 동료와 함께 문제를 마주한다. |
| 9 | 증거는 남아 있다 | 진실 직면 / 기록·제보 | 증거를 삭제하지 않고 기록을 확인 | 사건 기록을 저장/공유/신고한다. |
| 10 | 자동응답 | 자기 상실 / 시스템 동화 배드엔딩 | 시스템 명령에 계속 순응 | 생각하기 전에 “네, 알겠습니다”라고 답한다. |
| 11 | 미처리 감정 보관소 | 내면 직면 / 상징 | 꿈의 규칙을 이해하고 B1에 도달 | 마지막 입력어에 따라 책임 거절, 기록, 연락, 병가 등이 갈린다. |
| 12 | 오늘도 같은 꿈 | 완전 회피 / 루프 배드엔딩 | 동료·책임·진실·자기보호 모두 외면 | 출근하고 엘리베이터 층수가 `B1`로 바뀐다. |
| 13 | 처음으로 도움을 요청했다 | 자기보호 / 희망 | 혼자 해결하려다 실패하고 도움 필요성을 인식 | 동료/가족/상담 창구에 “힘들다”고 보낸다. |
| 14 | 문밖의 아침 | 조용한 해방 / 굿엔딩 | 진실 직면, 동료 배려, 부당 책임 거절 | 선택한 물건에 따라 현실 행동이 바뀐다. |
| 15 | 꿈이라서 다행이었다 | 메인 굿엔딩 후보 / 종합형 | 중요한 진실을 마주하고 현실 선택을 회피하지 않음 | 회피 대신 연락, 병가, 초안, 기록, 거절 중 하나를 실행한다. |

## 문체와 연출 지침 후보

- 과하게 설명하지 않는다.
- 감정은 해설보다 장면, 메시지, 손동작, 커서, 알림, 출근길 같은 구체물로 보여준다.
- 꿈속 장면은 반복, 왜곡, 압축, 상징을 사용한다.
- 현실 장면은 담백하게 쓴다.
- 시스템 메시지는 차갑고 간결하게 쓴다.
- TUI/터미널 기반 연출을 적극 활용한다.

TUI 문구 예시:

```text
SYSTEM: 출근을 환영합니다.
SYSTEM: 남은 업무를 불러오는 중...
SYSTEM: 사용자의 이름을 확인할 수 없습니다.
SYSTEM: 응답하지 않은 메시지 37개.
SYSTEM: 정말 나가시겠습니까?
SYSTEM: 이 작업은 되돌릴 수 없습니다.
SYSTEM: 무엇을 두고 나가시겠습니까?
```

현실 메시지 예시:

```text
{동료A}: 오늘 같이 들어가 줄 수 있어?
{동료A}: 혹시 어제 일, 너도 이상하다고 생각했어?
{상사B}: 어제 일은 네가 좀 덮어줘야겠다.
{상사B}: 오늘도 네가 처리해줘.
{주인공}: 아니요. 그건 제 일이 아닙니다.
{주인공}: 나 오늘 병가 낼게.
{주인공}: 출근 전에 잠깐 얘기하자.
```

## 품질 체크리스트 후보

나중에 엔딩 초안을 작성하거나 구현할 때 다음을 확인한다.

### 꿈 엔딩 검증

- “다 꿈이었다”로 플레이 경험을 무효화하지 않는가?
- 꿈속 선택이 깨어난 뒤 현실 행동에 반영되는가?
- 깨어난 뒤 주인공이 이전과 다르게 행동하는가?
- 달라지지 않는 경우, 그것이 명확한 배드엔딩으로 작동하는가?
- 꿈이라는 사실이 반전이 아니라 해석으로 작동하는가?

### 후일담 검증

- 깨어난 뒤 현실의 재시험 장면이 있는가?
- 후일담이 장황한 설명으로 흐르지 않는가?
- 플레이어가 “그래서 현실에서는?”이라는 질문의 답을 받는가?
- 마지막 장면이 행동으로 끝나는가?
- 엔딩명과 마지막 한 줄이 감정적으로 맞물리는가?

### 플레이어 감정 검증

- 플레이어가 허탈함보다 안도감, 긴장감, 여운, 재선택의 감각을 느끼는가?
- 플레이어가 자신의 선택이 엔딩의 색을 바꿨다고 느끼는가?
- 플레이어가 “속았다”가 아니라 “다시 읽힌다”고 느끼는가?

## 주의점

- 이 아이디어는 확정 요구사항이 아니라 후보 입력이다.
- 기존 `docs/content/Ending_List.md`의 13개 YAML 엔딩과 즉시 실패 엔딩을 당장 대체하지 않는다.
- 꿈 엔딩을 전체 메인 구조로 채택하면 현재 스토리/엔딩 문서와 큰 충돌이 생길 수 있으므로, 먼저 설계 문서에서 범위를 결정해야 한다.
- 공개 문서에는 실제 사용자의 사적 메모, 실제 회사/직원/위치, 민감한 현실 정보가 들어가면 안 된다.
- 현실 연결 엔딩과 결합할 경우, `docs/story/Reality_Link.md`의 공개/비공개 경계 원칙을 따라야 한다.
- “병가”, “사직서”, “제보”, “상담” 같은 현실 행동은 게임 내 캐릭터의 서사 선택으로만 다루고, 실제 사용자에게 직접 행동 지시처럼 보이지 않게 문맥을 분명히 한다.
- 배드엔딩도 후일담이 필요하다. 단순히 “아무 일도 없었다”가 아니라, 바뀌지 않았다는 사실이 장면으로 드러나야 한다.

## 설계자에게 물어볼 질문

- 이 꿈 엔딩 구조는 특정 스토리팩 전용인가, 전체 게임 엔딩 체계에 추가할 공통 구조인가?
- “깨어난 뒤 마지막 선택”을 실제 플레이 가능한 추가 턴으로 구현할 것인가, 엔딩 텍스트 내부의 분기/판정으로만 처리할 것인가?
- 현재 엔딩 타입(`escape`, `truth`, `conquest`, `hidden`, `failure`)에 `dream` 또는 `epilogue` 타입을 추가할 필요가 있는가?
