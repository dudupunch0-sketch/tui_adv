# UI 규칙

## 목적

`escape from the office`의 TUI는 일반 게임 HUD가 아니라, 이상한 사내 격리 대응 유틸리티처럼 보여야 한다.
플레이어가 믿는 대상은 캐릭터가 아니라 터미널, 사내망, 로그, 시스템 메시지다.
따라서 UI는 정보를 전달하면서도 다음 감각을 계속 유지해야 한다.

> 이 화면은 정말 나를 돕고 있는가?

## 핵심 원칙

1. 터미널은 중립적인 창이 아니라 세계관의 일부다.
2. 상태와 선택지는 게임 UI처럼 보이기보다 사내 시스템/진단 로그처럼 보여야 한다.
3. 글리치와 오염은 공포 장치이지만, 조작을 망치는 장치가 되어서는 안 된다.
4. 실제 회사 데이터를 읽는 느낌은 줄 수 있지만, 실제 민감정보를 읽거나 노출하지 않는다.
5. 모든 왜곡 연출은 테스트와 재현을 위해 시드 또는 명시적 상태에 의해 제어되어야 한다.

## 일반 게임 UI 회피

피해야 할 표현:

- 노골적인 HP bar
- 레벨, 경험치
- 퀘스트 마커
- 몬스터 이름/체력
- 전투 보상창
- 게임 외부에서만 이해되는 메뉴명

대신 사용할 표현:

- `[LOCAL STATUS]`
- `[격리 대응 유틸리티]`
- `[근태 시스템]`
- `[시설팀 공지]`
- `[workspace-sync]`
- `[NETWORK QUALITY]`
- 사내 메신저 알림
- 서버 로그
- 파일 목록
- 회의실 예약 패널
- 복합기 출력 로그

## 상태 표시 규칙

엔진 내부 상태는 숫자로 유지한다.

- health: 0-100
- sanity: 0-100
- battery: 0-100
- hunger: 0-100, 높을수록 위험
- thirst: 0-100, 높을수록 위험

하지만 기본 플레이 화면에서는 숫자를 노골적으로 드러내지 않는다.
상태는 사내 진단/로그처럼 표현한다.

예:

```text
[LOCAL STATUS]
신체 반응: 정상 범위
집중도: 불안정
단말기 전원: 43%
최근 식수 기록: 없음
권장 행동: 오래 머무르지 마십시오
```

또는:

```text
[격리 대응 유틸리티]
현재 위치: 7F 개발팀 사무실
사내망: 제한적 연결
생체 상태: 관찰 필요
정신 안정도: 불안정
단말기 전원: 43%
```

정확한 숫자는 다음 경우에만 보여준다.

- debug mode
- 상세 상태 화면
- 테스트용 로그
- 플레이어가 명확한 피드백을 받아야 하는 선택 결과

## 선택지 표시 규칙

선택지는 플레이어가 공정하게 판단할 수 있을 만큼 명확해야 한다.
단, 표현은 게임 메뉴보다 사내 시스템 명령처럼 보일 수 있다.

기본 예:

```text
[1] 사내 메신저 알림을 연다        [단말기 전원 소모]
[2] 화면을 끈다                    [집중도 회복 가능]
[3] 서버 로그를 추적한다            [필요: 단말기 전원 충분]
```

비활성 선택지는 숨기기보다 이유를 보여주는 것을 기본값으로 한다.
히든 루트 선택지만 조건 충족 전까지 숨길 수 있다.

## 글리치 등급

글리치는 세 등급으로 나눈다.

| 등급 | 목적 | 예시 | 입력 영향 |
|---|---|---|---|
| 장식 | 분위기 | 글자 깨짐, 상태 문구 흔들림 | 없음 |
| 신호 | 단서 암시 | 같은 단어 반복, 특정 파일명 오염 | 없음 또는 선택지 추가 |
| 개입 | 세계관적 강제 | 선택지 문구 변화, 입력 확인 왜곡 | 제한적으로 허용 |

1차 구현에서는 장식/신호까지만 안전하게 지원한다.
개입 등급은 후속 구현에서 별도 테스트를 통과한 장면에만 사용한다.

## 선택지 오염 규칙

허용되는 오염:

- 선택지 문구가 잠시 바뀌었다가 돌아옴
- 존재하지 않던 선택지가 보였다가 사라짐
- 선택지 옆 요구 조건 문구가 이상해짐
- 정신력 저하 상태에서 같은 문장이 반복됨

금지 또는 제한:

- 플레이어가 고른 선택을 아무 설명 없이 완전히 다른 선택으로 바꿈
- 되돌릴 수 없는 엔딩 선택을 왜곡함
- 핵심 튜토리얼 선택을 오염시킴
- 모든 선택지를 동시에 믿을 수 없게 만듦

선택지 오염은 다음 조건 중 하나 이상이 있을 때만 발생한다.

- sanity < 40
- thirst >= 70
- encounter가 `glitch` 또는 `reality_link` 태그를 가짐
- 특정 flag가 활성화됨
- 후반부 Act에서 공간 격리 규칙이 공개된 뒤

## 입력 왜곡 규칙

입력 왜곡은 가장 강한 연출이다.
남용하면 공포가 아니라 불공정함이 된다.

허용 조건:

- 특정 scripted encounter
- 실패해도 즉시 게임 오버가 아닌 장면
- 왜곡 후 로그나 단서로 원인을 암시할 수 있는 장면
- 플레이어가 같은 현상을 반복 관찰해 규칙을 추론할 수 있는 장면

금지 조건:

- 저장/종료 같은 메타 조작
- 접근성/도움말 입력
- 최종 엔딩 선택
- 현실 연결 최종 힌트 확인

예:

```text
> 2
입력 확인:
> 엘리베이터로 이동한다
```

이 장면 이후에는 반드시 다음과 같은 사후 단서를 남긴다.

```text
[LOG] 입력값과 처리값이 일치하지 않습니다. 관리자에게 문의하십시오.
```

## 시간 경과 연출 규칙

선택지를 띄운 상태에서 화면이 계속 변하는 연출은 압박감을 만들 수 있다.
하지만 읽을 시간을 빼앗으면 피로해진다.

기본값:

- 실제 턴은 플레이어 입력 전까지 진행하지 않는다.
- 대기 중 변화는 분위기 또는 시각적 변화에 그친다.
- 시간 제한 선택지는 명시적으로 표시한다.

예외:

- 후반부 추격/붕괴 장면
- 특정 히든 루트 퍼즐
- 사용자가 시간 제한 모드를 켠 경우

## 현실감 연출 안전 규칙

원칙:

> 실제로 읽지 말고, 읽은 것처럼 느끼게 한다.

기본 구현은 실제 회사 파일, 실제 문서, 실제 사내 시스템을 읽지 않는다.
현실감 있는 파일명, branch, commit, workspace, 로그는 더미 데이터로 제공한다.

허용:

- 공개 샘플 데이터
- 개발자가 직접 작성한 `private/` 또는 `*.local.*` allowlist 데이터
- 사용자가 명시적으로 opt-in한 안전한 값

금지:

- 실제 회사 문서 내용 읽기
- 홈 디렉토리나 workspace 임의 스캔
- 개인정보/직원명/조직도 노출
- 토큰, 인증 파일, 사내 URL 수집
- 위험하거나 잠긴 실제 위치 안내

## 접근성/가독성 규칙

- 색상만으로 정보를 전달하지 않는다.
- 중요한 선택지는 항상 번호를 고정한다.
- 글리치가 발생해도 도움말과 종료 조작은 정상 동작해야 한다.
- 한글 폭 정렬 문제를 테스트한다.
- 장식 글리치와 실제 단서 글리치는 패턴을 다르게 둔다.
- 플레이어가 선택 결과를 사후 로그로 이해할 수 있어야 한다.

## 구현 우선순위

1. 기본 상태/선택지/로그 UI를 안정적으로 만든다.
2. 숫자 상태를 사내 진단형 문구로 감싸는 표시 레이어를 만든다.
3. 장식 글리치와 신호 글리치를 optional presentation 데이터로 지원한다.
4. 선택지 오염은 제한된 encounter에서만 지원한다.
5. 입력 왜곡은 별도 테스트와 접근성 검토 후 추가한다.
