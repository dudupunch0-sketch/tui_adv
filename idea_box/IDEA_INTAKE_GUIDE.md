# 아이디어 수집 세션 운영 지침

이 문서는 사용자가 별도 Hermes 세션에서 계속 떠오르는 아이디어를 던질 때, agent가 그 내용을 `idea_box/inbox/`에 정리해 넣는 방식을 정의한다.

## 기본 전제

- 사용자가 이 세션에서 주는 내용은 즉시 현재 작업에 끼워 넣는 요구사항이 아니라, 나중에 순서대로 처리할 backlog 입력이다.
- `status: open`인 아이디어 문서는 나중에 설계/개발 agent가 Git 최초 반영 순서대로 반영/폐기/병합 판단해야 한다.
- agent는 아이디어를 과도하게 해석하거나 현재 프로젝트 우선순위에 끼워 넣으려 하지 않는다.
- 원문 의도와 뉘앙스를 최대한 보존하되, 나중에 읽기 쉽도록 구조화한다.

## 저장 위치

- 새 아이디어는 기본적으로 `idea_box/inbox/`에 저장한다.
- 아이디어 하나당 Markdown 파일 하나를 만든다.
- 서로 강하게 연결된 아이디어들은 하나의 파일에 묶어도 되지만, 서로 다른 개발 후보라면 파일을 나눈다.
- 파일명은 대략적인 생성일과 짧은 영문 slug를 사용한다.
- 같은 날짜 파일이 여러 개라 순서가 애매할 수 있으므로, Git에 올라간 뒤 `BACKLOG_ORDER.md`와 frontmatter의 order metadata를 갱신한다.

예시:

```text
idea_box/inbox/2026-05-21-distorted-save-slot.md
idea_box/inbox/2026-05-21-printer-terminal-ritual.md
```

## 권장 문서 구조

```md
---
status: open
created: YYYY-MM-DD
source: user
backlog_order:
git_added_at:
git_added_commit:
used_by:
done_at:
---

# 아이디어 제목

## 원문/요지
사용자가 말한 아이디어의 핵심을 원문 뉘앙스에 가깝게 요약한다.

## 핵심
나중에 설계자가 빠르게 이해할 수 있는 한두 문단 요약.

## 기대 효과
이 아이디어가 플레이 경험, 분위기, UI, 시스템, 스토리에 줄 수 있는 효과.

## 구현 후보
가능한 구현 방향. 확정 설계처럼 쓰지 말고 후보로 쓴다.

## 주의점
프로젝트 톤, fake-terminal 정체성, 현실 연결 안전 원칙, 우선순위 충돌 가능성 등을 적는다.

## 설계자에게 물어볼 질문
애매하거나 이상하거나 이해가 가지 않는 부분을 질문으로 남긴다.

## 처리 기록
- 아직 미사용.
```

## 정리 방식

1. 제목을 붙인다.
   - 사용자가 제목을 주지 않으면 agent가 임시 제목을 붙인다.
   - 제목은 나중에 검색하기 쉬워야 한다.

2. 원문 의도를 보존한다.
   - 사용자의 농담, 분위기, 불확실성도 필요하면 남긴다.
   - 너무 매끈한 기획 문서로 바꾸면서 이상한 확정을 만들지 않는다.

3. 구현 가능성을 분리한다.
   - `핵심`에는 아이디어 자체를 적는다.
   - `구현 후보`에는 가능한 구현 방법을 적는다.
   - 구현 후보는 언제나 후보이며, 실제 채택 여부는 나중의 설계/개발 판단에 맡긴다.

4. 질문을 남긴다.
   - 애매하거나 모순되거나 이상하게 느껴지는 부분은 임의로 해결하지 않는다.
   - `설계자에게 물어볼 질문`에 짧고 구체적인 질문으로 남긴다.
   - 예: “이 아이디어는 Python/Textual TUI 쪽인가, 브라우저 fake-TUI 쪽인가?”
   - 예: “이 효과는 실제 게임 규칙인가, 일회성 연출인가?”
   - 예: “현실 연결 힌트와 관련된다면 공개 문서에 둬도 안전한가?”

5. 즉시 현재 요구사항처럼 쓰지 않는다.
   - 아이디어 수집 시점에는 “구현 후보”, “가능성”, “검토할 수 있다” 같은 표현을 사용한다.
   - 하지만 `status: open`으로 저장된 뒤에는 backlog로 취급하며, 나중에 `BACKLOG_ORDER.md`의 순서대로 반영/폐기/병합 판단을 해야 한다.
   - 다른 agent가 현재 plan/todo를 먼저 수행해야 한다는 기존 우선순위를 유지한다.

## 하지 말 것

- 아이디어를 받자마자 현재 구현 작업 범위에 끼워 넣지 않는다.
- 불분명한 부분을 agent가 혼자 확정하지 않는다.
- 사적 메모, 실제 위치, 회사 기밀, 위험한 현실 행동을 공개 문서에 적지 않는다.
- `done`을 단순히 읽었다는 뜻으로 쓰지 않는다.
- 사용자가 아이디어를 던지는 도중 매번 과도한 확인 질문으로 흐름을 끊지 않는다. 필요한 질문은 문서에 남긴다.

## done 처리

이 수집 세션에서는 보통 새 아이디어를 `open` 상태로 둔다. `done` 처리는 나중에 설계/개발 agent가 실제로 다음 중 하나를 수행했을 때 한다.

- 설계 문서에 반영했다.
- 코드/데이터/테스트에 반영했다.
- 다른 아이디어나 계획에 병합했다.
- 프로젝트 방향과 맞지 않는다고 명시적으로 폐기했다.

처리할 때는 어디에 반영했는지, 또는 왜 폐기/병합했는지 기록한다.
