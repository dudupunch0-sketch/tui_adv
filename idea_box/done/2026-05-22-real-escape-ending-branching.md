# 현실 탈출 엔딩 분기 후보

## 원문/요지

사용자는 `idea_box/real_escape_ending_branch_ideas.md`에 “꿈이 아닌 엔딩”, 즉 모든 사건이 실제로 일어났고 플레이어가 현실의 결과를 감당하는 엔딩 분기 가이드를 추가했다.

원문 핵심 문장:

```text
탈출은 문을 나서는 순간 끝나는 것이 아니라,
문밖으로 무엇을 가져왔는지로 완성된다.
```

꿈 엔딩이 “깨어난 뒤 진짜 선택이 시작된다”라면, 현실 탈출 엔딩은 “나간 뒤 진짜 대가가 시작된다”에 가깝다.

이 문서는 원문을 나중의 설계/개발 agent가 바로 검토할 수 있도록 구조화한 아이디어 후보 문서다. 아직 확정 엔딩 시스템이나 구현 요구사항이 아니며, 기존 런타임 엔딩 목록을 즉시 대체하라는 뜻도 아니다.

## 핵심

현실 탈출 엔딩에서 탈출은 최종 보상이 아니라 엔딩 판정의 시작 조건이다.

좋은 현실 탈출 엔딩은 다음 질문을 남긴다.

```text
밖으로 나왔다.
그런데 누구를 데리고 나왔는가?
무엇을 증명할 수 있는가?
무엇을 버리고 나왔는가?
무엇이 아직 안에 남아 있는가?
```

핵심 감정:

- 생존
- 대가
- 책임
- 증거
- 후폭풍
- 회사의 반응
- 다음 이야기의 흔적

## 꿈 엔딩과의 대비

| 구분 | 꿈 엔딩 | 현실 탈출 엔딩 |
|---|---|---|
| 사건 성격 | 악몽, 내면의 시뮬레이션, 회귀성 경험 | 실제로 일어난 감금/이상현상/거래/실종/폭로 |
| 핵심 감정 | 안도, 두 번째 기회 | 생존, 대가, 책임, 후폭풍 |
| 엔딩 이후 | 깨어난 뒤 현실 선택이 중요 | 나온 뒤 현실의 결과 정산이 중요 |
| 플레이어 감정 | “꿈이라서 다행이다. 이제 다르게 선택할 수 있다.” | “나왔다. 하지만 이제 진짜 문제가 시작된다.” |
| 다음 플레이 영향 | 더 나은 선택의 회귀성 경험 | 세계에 남은 흔적, 로그, 경고, 추적, 기록 변화 |

## 기대 효과

- 기존 `escape`, `truth`, `hidden`, `failure`, `conquest` 계열 엔딩에 후일담/결과 정산 레이어를 추가할 수 있다.
- “탈출 성공 = 굿엔딩”이 되는 얕은 구조를 피하고, 플레이어 선택을 더 오래 남길 수 있다.
- 증거, 동료, 진실 공개, 회사와의 거래, 정체성 보존 같은 축을 통해 엔딩 색을 다양하게 만들 수 있다.
- TUI/fake-terminal 엔딩 후일담에 잘 맞는다. 뉴스, 회사 공지, 메신저 로그, 경찰/감사 진술서, 시스템 리포트, HR Core 로그를 사용할 수 있다.
- 이전 꿈 엔딩 아이디어와 쌍을 이룬다. 꿈 엔딩이 `재선택`이라면 현실 탈출 엔딩은 `결과 정산`이다.

## 구현 후보

### 1. 설계 문서 후보로 먼저 승격

바로 런타임 데이터에 넣기보다, 먼저 `docs/story/` 또는 `docs/design/` 아래에 “현실 탈출 엔딩 분기 설계” 문서로 승격하는 방식을 검토할 수 있다.

후보 문서명:

```text
docs/story/Real_Escape_Ending_Branching.md
# 또는
docs/design/Real_Escape_Ending_Branching.md
```

해당 설계 문서에서는 다음을 결정해야 한다.

- 현실 탈출 엔딩을 기존 엔딩 타입의 후일담 레이어로 둘지
- `real_escape`, `post_escape`, `aftermath` 같은 새 엔딩 타입을 추가할지
- 꿈 엔딩과 같은 스토리팩 안에서 병렬 분기로 둘지
- 다음 플레이 영향이 실제 저장 상태인지, 단순 텍스트/로그 해금인지

### 2. 현재 엔딩 데이터와 연결 후보

현재 공개 문서 기준 런타임 엔딩은 `docs/content/Ending_List.md`에 정리되어 있다. 현실 탈출 엔딩을 구현할 경우, 기존 엔딩을 즉시 삭제/대체하지 말고 다음 중 하나를 검토한다.

- 기존 `escape` 엔딩에 후일담 결과 정산을 추가한다.
- 기존 `truth` 엔딩과 결합해 증거/공개/감사 개시 분기를 만든다.
- `hidden` 현실 연결 엔딩과 결합할 때는 실제 민감정보를 공개 문서로 옮기지 않는다.
- “탈출 직전 마지막 선택”과 “탈출 후 후일담”을 분리한 2단 엔딩 구조를 둔다.
- 후일담 형식을 텍스트 블록, 시스템 리포트, 뉴스/공지 UI, 메신저 로그 중 하나로 선택한다.

주의: 이 아이디어는 아직 기존 `src` 엔딩 판정이나 YAML 스키마를 바꾸라는 지시가 아니다. 구현 전에 설계 결정이 필요하다.

## 권장 상태 변수 후보

원문에서 제안한 엔딩 설계용 판단 축이다. 실제 코드 변수로 확정된 것은 아니다.

```text
escaped: true | false
alive: true | false
rescued_count: number
rescued_key_npc: true | false
abandoned_key_npc: true | false
witness_count: number

has_evidence: true | false
evidence_level: 0 | 1 | 2 | 3
# 0 = 없음
# 1 = 정황 증거
# 2 = 내부 자료/녹취/로그 등 실질 증거
# 3 = 원본 로그, 핵심 파일, 다중 증언, 외부 전송 완료

evidence_uploaded: true | false
evidence_destroyed: true | false
evidence_traded: true | false

truth_action: none | silent | private_share | public_leak | official_report | company_trade | destroy
company_response: denial | retaliation | negotiation | audit | collapse_hint | recruitment

self_sacrifice: true | false
identity_preserved: true | false
employee_record_exists: true | false
company_assimilation: true | false

post_escape_risk: low | medium | high
next_play_unlock: none | clue | command | altered_intro | hidden_log | npc_memory
```

## 엔딩 선택 우선순위 후보

여러 조건이 동시에 충족될 수 있으므로, 엔딩 선택에는 우선순위가 필요하다.

1. `self_sacrifice = true`이면 자기희생/잔류 계열 우선.
2. `company_assimilation = true` 또는 `evidence_traded = true`이면 거래/동화 계열 우선.
3. `evidence_level >= 3`, `rescued_key_npc = true`, `truth_action`이 공개/공식 신고라면 트루엔딩 계열.
4. `evidence_level >= 2`이고 혼자 탈출했다면 증거 엔딩 계열.
5. `rescued_count >= 1`이고 증거가 부족하다면 동료 구출 엔딩 계열.
6. `escaped = true`이나 증거도 동료도 없다면 생존/불신 엔딩 계열.
7. `escaped = true`이나 `identity_preserved = false`라면 정체성 상실 계열을 섞는다.
8. `escaped = false`이면 잔류/실종/희생/포획 엔딩 계열.

우선순위는 절대 규칙이 아니라 충돌 방지용 가이드다. 특정 스토리팩의 핵심 테마가 있으면 그 테마를 우선해야 한다.

## 핵심 분기 축

### 생존 축

- 살아서 나감
- 다쳤지만 나감
- 정신적/사회적 정체성을 잃고 나감
- 몸은 나갔지만 회사 시스템에 남음
- 타인을 내보내고 자신은 남음
- 실패해서 안에 갇힘

### 증거 축

- 증거 없음: 아무도 믿어주지 않음
- 정황 증거: 루머, 의혹, 개인 진술 수준
- 실질 증거: 로그, 녹취, 파일, 사진, 내부 문건
- 결정적 증거: 원본 서버 로그, 다중 증언, 외부 전송 완료, 삭제 불가능한 공개 기록

증거가 있다고 반드시 굿엔딩은 아니다. 증거를 어떻게 쓰는지가 더 중요할 수 있다.

### 동료 축

- 동료를 구함
- 동료를 버림
- 동료를 속임
- 동료가 플레이어를 배신함
- 동료가 플레이어를 증언함
- 동료가 플레이어 대신 남음
- 동료가 회사 쪽 사람이었음

### 진실 공개 축

- 공개하지 않음
- 특정 동료에게만 공유
- 익명 게시판에 업로드
- 언론에 제보
- 경찰/감사기관에 신고
- 회사와 협상
- 인사팀/상부에 넘김
- 서버와 함께 파괴

### 회사 동화 축

플레이어가 살아남고 보상도 받았지만, 사실상 회사 시스템의 일부가 되는 배드엔딩 축이다.

예시:

- 사원증 권한이 상승한다.
- 임원 전용 엘리베이터가 열린다.
- 플레이어가 은폐 업무를 맡는다.
- 회사가 플레이어에게 “보호”를 제안한다.
- 실종된 동료의 자리를 플레이어가 차지한다.
- 플레이어가 다음 희생자를 안내하는 역할이 된다.

핵심 감정:

```text
탈출한 것이 아니라 승격됐다.
그리고 그게 더 무섭다.
```

### 정체성 보존 축

- 출입 기록이 없다.
- 사원번호가 말소된다.
- 메일 계정이 삭제된다.
- 모든 CCTV에서 플레이어만 누락된다.
- 인사 시스템에서 플레이어가 `퇴사자`가 아니라 `미등록 자산`으로 분류된다.
- 동료는 플레이어를 기억하지만 회사 기록은 부정한다.
- 회사 기록은 남아 있지만 플레이어가 자신의 이름을 기억하지 못한다.

## 후일담 형식 후보

| 형식 | 잘 맞는 엔딩 |
|---|---|
| 뉴스 기사 | 공개 폭로, 감사, 사건 확대 |
| 회사 공지 | 은폐, 부인, 법적 대응, 사내 통제 |
| 메신저 로그 | 동료 구출, 죄책감, 관계 회복 |
| 경찰/감사 진술서 | 현실성, 공식성, 불신 |
| 시스템 리포트 | TUI 스타일, 결과 정산, 상태 변수 표시 |
| 사원 시스템 로그 | 정체성 상실, 회사 동화, 추적 |

시스템 리포트 예시:

```text
[POST-ESCAPE REPORT]

survivor_count: 2
evidence_uploaded: true
internal_audit: initiated
company_response: denial
employee_status: terminated
risk_level: ongoing

ENDING: 감사 개시
```

HR 로그 예시:

```text
[HR CORE]

employee_id: NOT FOUND
access_level: revoked
classification: unregistered_asset
recovery_team: dispatched
```

## 엔딩 작성 포맷 후보

나중에 콘텐츠 agent가 엔딩 초안을 만들 때는 각 엔딩을 다음 항목으로 작성하면 좋다.

```text
엔딩명
조건
마지막 장면
후일담
감정
다음 플레이 영향 선택 사항
```

`마지막 장면`에는 탈출 직전/탈출 순간에 플레이어가 무엇을 가지고 나오고 무엇을 두고 나오는지 명확히 보여준다.

`후일담`에는 뉴스, 메일, 메신저, 시스템 리포트, 병원 장면, 경찰 진술서 등으로 탈출 이후의 현실 결과를 보여준다.

## 대표 엔딩 후보 목록

| 번호 | 엔딩명 | 핵심 조건 후보 | 핵심 감정 | 다음 플레이 영향 후보 |
|---:|---|---|---|---|
| 1 | 정문 밖 | 살아서 탈출했지만 증거도 동료도 확보하지 못함 | 살았지만 아무도 믿어주지 않음 | 첫 화면에 `incident_report: rejected` 같은 로그 |
| 2 | 업로드 완료 | 핵심 증거를 외부 전송하고 혼자 탈출 | 이겼지만 이제 쫓김 | 새 명령어/로그 해금, 추적 메시지 |
| 3 | 두 명 이상 퇴근 가능 | 동료 1명 이상과 함께 탈출, 증거는 불완전 | 회사는 무너지지 않았지만 증인이 생김 | 동료 기억/대사 변화 |
| 4 | 감사 개시 | 핵심 증거 + 증인 + 공식/신뢰 채널 전달 | 제대로 해냈지만 싸움은 시작됨 | 감사 로그, 외부기관 메시지 |
| 5 | 임원 엘리베이터 | 증거를 회사와 거래하고 동료를 버리거나 침묵시킴 | 탈출이 아니라 승격됨 | 권한 상승, 불쾌한 새 메뉴 |
| 6 | 없었던 야근 | 일부 증거가 있지만 공개하지 않고 회사가 은폐 | 안전하지만 침묵의 대가가 남음 | 사내공지/익명글로 불안 암시 |
| 7 | 서버실에 남은 사람 | 동료나 증거를 밖으로 보내고 플레이어는 남음 | 나는 못 나갔지만 문은 열었음 | 다음 회차 숨겨진 서버 로그 |
| 8 | 출입기록 없음 | 물리적으로 탈출했지만 시스템상 존재가 지워짐 | 나왔지만 내가 있었다는 증거가 없음 | `employee_id: NOT FOUND` 상태 |
| 9 | 회수팀 호출 | 증거를 가지고 탈출했고 회사가 적극 추적 | 탈출했지만 퇴근은 승인되지 않음 | 회수팀 알림/추적 이벤트 |
| 10 | 증인 보호 | 증거는 부족하지만 다수 증언이 확보됨 | 침묵이 완벽하지 않게 됨 | 증언자 NPC 기억 유지 |
| 11 | 퇴사 처리 통지 | 탈출 후 회사가 보복으로 플레이어를 공식 절단 | 나가는 방식도 회사가 정함 | 퇴사자/미등록 상태 시작 |
| 12 | 재입사 제안 | 회사 비밀을 일부 이해하고 회사가 제거 대신 회유 | 회사가 아직 나를 후보자로 봄 | 제안 메일/임원 초대 |
| 13 | 서버와 함께 묻다 | 증거를 확보했지만 공개하지 않고 파괴 | 회사를 상처 입혔지만 진실도 죽임 | 결손 로그/삭제된 증거 흔적 |
| 14 | 마지막 퇴근자 | 동료를 구하지 못하고 마지막으로 혼자 나옴 | 혼자 살아남은 죄책감 | 빈 좌석/부재자 흔적 강화 |
| 15 | 문밖으로 가져간 것 | 증거, 동료, 정체성, 양심 중 일부를 지킴 | 끝나지 않았지만 빈손은 아님 | 대표 노멀/굿엔딩용 흔적 |

## 다음 플레이 영향 후보

현실 탈출 엔딩의 다음 플레이 영향은 “회귀”보다 “흔적”이 좋다.

좋은 후보:

- 새 로그 해금
- 숨겨진 명령어 해금
- 특정 NPC의 대사 변화
- 첫 화면의 아주 짧은 오류 메시지
- 특정 문서의 주석 추가
- 이전 엔딩명이 시스템 로그에 남음
- 일부 선택지가 더 일찍 등장
- 플레이어가 이전 회차의 물건을 꿈처럼 떠올림

피해야 할 후보:

- 다음 플레이를 하지 않으면 엔딩이 이해되지 않음
- 이번 엔딩이 사실상 미완성으로 느껴짐
- 다음 플레이 보상이 너무 커서 특정 엔딩만 정답처럼 됨
- 모든 엔딩을 강제로 메타 루프로 묶어 현실 탈출의 무게를 약화함

예시:

```text
previous_exit: uploaded_complete
unlocked_command: trace_origin
initial_warning: 회사는 아직 당신의 퇴근을 승인하지 않았습니다.
```

## 주의점

- 이 아이디어는 확정 요구사항이 아니라 후보 입력이다.
- 기존 `docs/content/Ending_List.md`의 런타임 엔딩을 당장 대체하지 않는다.
- 탈출 후 모든 문제가 즉시 해결되면 현실감이 떨어진다. 회사는 너무 쉽게 무너지지 않아야 한다.
- 증거가 없어도 모두가 믿어주면 플레이어 선택의 가치가 약해진다.
- 동료를 버려도 아무 대가가 없으면 윤리적 선택이 무의미해진다.
- 거래 엔딩은 단순 성공이 아니라 불쾌한 보상이어야 한다.
- 다음 플레이 영향은 보너스/흔적이어야 하며, 현재 엔딩의 감정적 완결성을 해치면 안 된다.
- 공개 문서와 런타임 콘텐츠에는 실제 사용자의 사적 메모, 실제 회사/직원/위치, 민감한 현실 정보를 넣지 않는다.
- 현실 연결 히든 엔딩과 결합할 경우 `docs/story/Reality_Link.md`의 공개/비공개 경계 원칙을 따라야 한다.

## 품질 체크리스트 후보

나중에 엔딩 초안을 작성하거나 구현할 때 다음을 확인한다.

- 이 엔딩은 꿈이 아니라 실제 사건으로 느껴지는가?
- 플레이어가 무엇을 들고 나왔는지 명확한가?
- 플레이어가 무엇을 버리고 나왔는지 명확한가?
- 동료, 증거, 진실 공개, 회사와의 관계 중 최소 하나가 반영되었는가?
- 후일담에서 탈출 이후의 현실 결과가 드러나는가?
- 모든 문제가 너무 쉽게 해결되지 않는가?
- 반대로, 플레이어의 노력이 완전히 무의미해지지도 않는가?
- 엔딩명만 봐도 감정이 떠오르는가?
- 다음 플레이 영향이 있다면 현재 엔딩의 완결성을 해치지 않는가?
- 플레이어가 “나왔다. 하지만 이제부터가 진짜다”라고 느낄 수 있는가?

## 설계자에게 물어볼 질문

- 현실 탈출 엔딩은 기존 `escape` 엔딩의 후일담 레이어인가, 별도 엔딩 타입인가?
- 꿈 엔딩 후보와 같은 스토리팩 안에서 어떻게 병렬 배치할 것인가?
- “탈출 직전 마지막 선택”을 실제 플레이 선택지로 구현할 것인가, 엔딩 판정 내부 조건으로 처리할 것인가?
- `evidence_level`, `truth_action`, `identity_preserved` 같은 변수를 실제 런타임 상태로 둘 것인가, 콘텐츠 작성용 판단 축으로만 둘 것인가?
- 대표 엔딩 15개 중 1차 구현 후보는 무엇인가?
- 현재 게임의 기본 탈출 엔딩(`escape_commute`, `escape_rooftop_signal`, `escape_parking_lot`, `escape_lobby_revolving_door`)에 어떤 후일담을 붙일 수 있는가?
- 동료/증거/회사 보복을 실제 NPC/아이템/플래그 시스템과 어떻게 연결할 것인가?
- 다음 플레이 영향은 저장 상태를 바꿀 것인가, 해금 로그/문구 정도로 제한할 것인가?
- 현실 연결 히든 엔딩과 결합할 경우 공개 문서와 로컬 비공개 secret의 경계를 어떻게 유지할 것인가?
