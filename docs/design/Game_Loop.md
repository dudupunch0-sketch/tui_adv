# 게임 루프 설계

## 목적

게임 루프는 플레이어가 매 턴 무엇을 보고, 무엇을 선택하고, 어떤 결과를 받는지 정의한다.
`escape from the office`는 복잡한 조작보다 “읽고 판단하고 감당하는 선택”에 집중한다.

## 기본 턴 구조

한 턴은 하나의 의미 있는 행동이다.

예:

- 위치 이동
- 인카운터 선택지 선택
- 아이템 사용
- 사내망 확인
- 휴식
- 문/패널/복합기 등 상호작용

기본 순서:

1. 현재 상태 표시
2. 현재 위치 표시
3. 현재 위치에서 가능한 이동/행동 후보 생성
4. 위치/조건 기반 인카운터 선택
5. 이벤트 설명과 선택지 표시
6. 플레이어 입력 처리
7. 선택 비용 적용
8. 성공/실패/확정 결과 계산
9. 상태/아이템/단서/플래그/위치 갱신
10. 로그 추가
11. 턴 경과 효과 적용
12. 엔딩 조건 확인
13. 다음 턴으로 진행

## 턴 경과 효과

기본 턴 경과:

- 허기 +1
- 갈증 +2

상황별 추가 변화:

- 어두운 구역에서 손전등 사용: 배터리 감소
- 위험한 이동: 체력 감소 가능
- 괴현상 접촉: 정신력 감소 가능
- 물/음식 섭취: 허기/갈증 감소
- 휴식: 체력/정신력 일부 회복, 대신 허기/갈증 증가

## 인카운터 선택 방식

1차 구현은 단순하게 시작한다.

우선순위:

1. 강제 이벤트
2. 플래그 조건 이벤트
3. 현재 위치 고유 이벤트
4. 공통 랜덤 이벤트
5. 아무 일도 없음 또는 이동 선택

예:

- `printer_secret_started` 플래그가 있고 복합기 구역에 있으면 후속 출력물 이벤트 우선
- 탕비실에서는 음식/물/커피머신 이벤트 확률 증가
- 비상계단에서는 탈출 루트 이벤트 또는 발소리 이벤트 확률 증가

## 선택지 구조

각 선택지는 다음 요소를 가진다.

- 표시 텍스트
- 필요 조건
- 비용
- 성공 조건 또는 확률
- 성공 결과
- 실패 결과
- 로그 문구
- 후속 플래그

예:

```text
1. 메시지를 확인한다
   조건: 배터리 >= 3
   비용: 배터리 -3
   결과: 단서 획득 또는 정신력 감소
```

## 1차 구현 기준

현재 도메인 엔진은 다음 순수 로직을 먼저 제공한다.

- `Encounter`: 위치/재난/기록 조건과 선택지 목록을 가진 이벤트
- `Choice`: 표시 텍스트, 조건, 비용, 결과, 선택적 능력치 판정을 가진 선택지
- `Conditions`: 위치, 재난 타입, 아이템, 플래그, 자원 최소/최대, 능력치 최소 조건 검사
- `Outcome`: 자원 변화, 아이템/단서/플래그 변화, 위치 이동, 위험도, 로그 적용
- `AbilityCheck`: 2d6 + 능력치로 성공/실패 결과를 나누는 판정
- `eligible_encounters`: 현재 상태에서 등장 가능한 인카운터 목록 생성
- `select_encounter`: 등장 가능한 인카운터 중 weight 기반 랜덤 선택
- 선택지 해결 시 비용과 결과를 적용한 뒤 한 턴을 진행한다.
- 반복 불가 인카운터는 `seen_encounters`에 있으면 다시 등장하지 않는다.

1차 샘플 인카운터는 `퇴사자의 메신저`, `복합기가 혼자 출력한다`, `탕비실 커피머신`, `서버실 문 앞 무전기`이다.
런타임 기본 인카운터와 루트 엔딩은 `src/tui_adv/data/encounters.yaml`, `src/tui_adv/data/endings.yaml`에서 로드한다.
CLI smoke는 새 게임 상태에서 현재 위치 인카운터를 선택해 제목, 본문, 선택지, 비용/판정 정보를 함께 출력한다.
`--choice <번호>`를 함께 주면 현재 표시된 선택지를 1부터 시작하는 번호로 실행하고, 결과 로그/자원 변화/다음 턴 상태를 출력한다.
`--location <id>`와 반복 가능한 `--flag <id>`를 함께 주면 탈출/실패 같은 결정적 smoke 상태를 만들 수 있다.
선택지 실행 결과는 판정 메타데이터, 새 로그, 자원 변화를 포맷터가 렌더링한다.
엔딩 판정은 턴 종료 후 실행하며, 1차 구현은 공간 왜곡 퍼즐 성공 시 `escape_commute`, 잘못된 계단 선택 시 `game_over_spatial_collapse`, 서버실 앞 무전기 장악 시 `conquest_broadcast_channel`을 출력한다.
Textual 수직 슬라이스는 `--tui-smoke`로 같은 상태/인카운터/선택지/로그 레이아웃을 비대화형으로 검증하고, `--tui`는 Textual 설치 환경에서 숫자 키 선택을 처리한다.

## 선택지 유형

### 확정 선택지

선택하면 항상 같은 결과를 낸다.

예:

- 생수를 마신다: 갈증 -30
- 복도로 돌아간다: 위치 이동

### 조건 선택지

아이템, 플래그, 상태 수치, 능력치가 있어야 선택 가능하다.

예:

- 사원증을 찍는다
- 손전등으로 안쪽을 비춘다
- 구겨진 출력물을 복합기에 다시 넣는다
- `[인터페이스] 알림 지연 시간을 역추적한다`

### 능력치 판정 선택지

선택 자체는 가능하지만, 결과가 `2d6 + 능력치 >= 난이도`로 갈린다.
능력치가 높은 플레이어는 같은 상황에서도 더 많은 단서를 얻거나, 실패 피해를 줄일 수 있다.

예:

- `[인터페이스] 패킷 지연을 읽는다`: 성공 시 숨은 라우팅 단서, 실패 시 정신력 피해와 위험도 증가
- `[논리] 회의록의 시점 모순을 짚는다`: 성공 시 진실 플래그, 실패 시 무의미한 문장만 남음
- `[의지] 방송을 끝까지 듣는다`: 성공 시 단서, 실패 시 정신력 피해

### 위험 선택지

성공/실패가 나뉜다.

예:

- 비상계단 어둠 속으로 내려간다
- 서버실 앞 패널을 강제로 연다
- 회의실 안의 목소리를 녹음한다

### 히든 선택지

특정 단서나 플래그가 있을 때만 나타난다.

예:

- 토너 카트리지를 확인한다
- 예약 패널의 숨겨진 로그를 연다
- 냉장고 자석 아래를 확인한다

## 성공/실패 기준

1차 구현은 `AbilityCheck`로 2d6 판정을 제공한다.

기준:

- 판정식: `1d6 + 1d6 + ability >= difficulty`
- 능력치 범위: 0-6
- 판정은 seed/turn/encounter/choice 기반으로 재현 가능해야 한다.
- 테스트에서는 RNG를 주입해 성공/실패를 고정한다.

가능한 보조 기준:

- 상태 수치 임계치
- 아이템 보유 여부
- 플래그 보유 여부
- 고정 확률
- 위험도 수치

예:

```text
성공 조건:
- 정신력 >= 40이면 성공
- 정신력 < 40이면 실패 또는 왜곡 결과
```

또는:

```text
성공 확률:
- 기본 70%
- 체력 40 미만이면 -20%
- 허기 70 이상이면 -10%
```

## 위험도

위험도는 회사 전체 또는 현재 구역의 불안정성을 나타낸다.

증가 조건:

- 큰 소리 발생
- 오래 머무름
- 사내 방송/보안 시스템 조작
- 괴현상에 직접 접촉
- 금지 구역 진입
- 반복 공간 규칙 위반

감소 조건:

- 문 잠금
- 은신
- 안전 구역 복귀
- 보안 시스템 일부 장악
- 괴현상 규칙 파악

1차 구현에서는 위험도를 0-5 또는 0-10 범위로 단순화한다.

## 사내망과 제한 인터넷

외부 인터넷은 기본적으로 불안정하거나 차단된다.
하지만 사내망은 간헐적으로 작동한다.

사내망으로 가능한 것:

- 메신저 확인
- 회의실 예약 패널 확인
- 사내공지 확인
- 서버/전산 로그 일부 열람
- 보안 시스템 상태 확인
- 누군가의 메시지 수신

사내망 행동은 대부분 배터리를 소모한다.

## 로그

로그는 최근 행동과 결과를 보여준다.

일반 로그:

- “탕비실로 이동했다.”
- “생수를 마셨다. 갈증 -30.”
- “사내망 알림이 도착했다.”

괴현상 로그:

- “회의실 B는 예약되어 있지 않다. 방금 전까지는.”
- “복합기가 출력한 종이에는 아직 쓰지 않은 당신의 답변이 적혀 있다.”

정신력 낮음 로그:

- “누군가 당신의 이름으로 회의에 참석했다.”
- “로그가 한 줄 더 있었다. 읽는 순간 사라졌다.”

## 엔딩 체크

매 턴 마지막에 엔딩 조건을 확인한다.

우선순위 초안:

1. 즉시 실패: 체력 0, 정신력 0
2. 특수 실패: 갈증/허기 한계와 특정 위험 조건
3. 히든 현실 연결 루트
4. 진실 루트
5. 정복 루트
6. 탈출 루트

우선순위는 추후 조정 가능하다.
히든/진실/정복/탈출 조건이 동시에 만족될 수 있으므로, 엔딩 판정 규칙은 명확해야 한다.

## 1차 플레이 흐름 예시

1. 내 자리에서 시작한다.
2. 사내망 메신저 알림이 온다.
3. 배터리를 써서 확인할지 무시할지 선택한다.
4. 개발팀 사무실 또는 복도로 이동한다.
5. 탕비실에서 물/음식을 얻거나 이상한 커피머신을 조사한다.
6. 복합기 구역에서 구겨진 출력물을 얻는다.
7. 회의실 예약 패널에서 존재하지 않는 회의 정보를 본다.
8. 서버실 앞에서 사내망 로그 단서를 얻는다.
9. 비상계단을 통해 탈출하거나, 히든 현실 연결 힌트를 얻는다.

## 구현 원칙

- 게임 엔진은 TUI에 의존하지 않는다.
- 모든 결과는 테스트 가능한 순수 로직으로 처리한다.
- 인카운터와 선택지는 나중에 데이터 파일로 분리할 수 있게 설계한다.
- 랜덤은 시드를 받을 수 있게 해서 테스트 재현성을 확보한다.
