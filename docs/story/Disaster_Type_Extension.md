# 재난 타입 확장 규칙

이 문서는 `escape from the office`의 기본 재난인 `unknown_isolation` 이후 새 재난 타입을 추가할 때 지켜야 하는 공개-safe 설계 규칙과 후보 백로그를 정의한다.
현재 런타임은 `GameState.disaster_type`, save JSON의 `disaster_type`, YAML 조건의 `conditions.disaster_types`를 이미 읽지만, 플레이어가 시작 시 재난 타입을 고르는 UI나 별도 재난 타입 registry는 아직 없다.
따라서 이 문서는 구현 완료 기능 목록이 아니라 다음 콘텐츠 확장을 안전하게 쪼개기 위한 설계 계약이다.

## 현재 기준 타입

| id | 상태 | 핵심 판타지 | 공개-safe 경계 |
|---|---|---|---|
| `unknown_isolation` | 현재 기본값 | 회사 격리 프로토콜이 사람과 공간을 업무 로그처럼 복구하려다 사무실을 재시도 가능한 작업 단위로 오인한다. | 실제 회사명, 실제 시스템명, 실제 보안 구조, 현실 최종 좌표를 쓰지 않는다. |

`unknown_isolation`은 아직 “불명 재난”처럼 보이다가 진실 루트에서 격리 프로토콜로 해석되는 타입이다.
새 타입을 추가해도 이 타입은 기본 시작점과 회귀 기준으로 남긴다.

## 확장 원칙

1. 새 타입은 “스킨”이 아니라 플레이 압박을 바꿔야 한다.
   - 최소 하나의 자원 압박 축, 하나의 위치/루트 우선순위, 하나의 UI 문구 패턴이 달라져야 한다.
2. 기존 루트를 모두 복제하지 않는다.
   - 탈출, 정복, 진실, 히든 현실 연결 중 1~2개 루트만 타입별 변주를 먼저 만든다.
3. 공개 저장소에는 안전한 은유만 둔다.
   - 실제 회사 사고, 실제 내부 시스템, 실제 공간 구조를 추정할 수 있는 문장은 금지한다.
4. 타입별 단서는 단계적으로 드러낸다.
   - 시작 화면은 모호하게, 중반 단서는 패턴으로, 엔딩은 해석으로 정리한다.
5. 데이터 hook을 먼저 쓰고 엔진 분기는 늦게 만든다.
   - 가능한 한 `conditions.disaster_types`, 위치 태그, 플래그, 자원 조건으로 표현한다.
   - 새 Python/Rust/TypeScript 분기는 데이터 hook으로 불가능한 경우에만 추가한다.
6. 브라우저 fake-TUI와 Rust content runner parity를 깨지 않는다.
   - 타입별 콘텐츠를 추가하면 YAML export, content bundle, web generated JSON, Rust fixture를 같이 갱신한다.

## 런타임 연결 계약

현재 코드와 문서가 보장하는 연결점은 다음이다.

| 연결점 | 현재 상태 | 새 타입 추가 시 사용법 |
|---|---|---|
| `GameState.disaster_type` | 기본값 `unknown_isolation` | 새 게임 생성 또는 테스트 fixture에서 타입 id를 주입한다. |
| save JSON `disaster_type` | 저장/로드됨 | 오래된 save와 호환되게 기본값은 유지한다. |
| `conditions.disaster_types` | 인카운터/선택지/엔딩 조건에서 읽음 | 타입 전용 이벤트를 YAML 조건으로 제한한다. |
| CLI/TUI 표시 | CLI smoke가 재난 id를 표시함 | 사용자 표시명 registry가 생기기 전에는 id가 노출될 수 있음을 감안한다. |
| 시작 UI 타입 선택 | 아직 없음 | 첫 구현 slice에서는 smoke/fixture 주입으로 검증하고, UI 선택은 별도 slice로 둔다. |

새 타입을 추가하는 첫 PR은 다음 중 하나에만 집중한다.

- 설계/백로그 문서화
- 타입 id를 seedable smoke에 주입하는 최소 runtime slice
- 타입 전용 YAML 인카운터 1개와 엔딩 1개
- Textual 시작 화면의 타입 선택 UX

한 PR에서 네 가지를 모두 처리하지 않는다.

## 타입 id 규칙

- 소문자 snake_case를 사용한다.
- 현상보다 플레이 규칙을 드러내는 이름을 우선한다.
- 회사 실명, 실제 프로젝트명, 실제 장애명, 내부 시스템명을 연상시키는 이름은 쓰지 않는다.
- 예: `blackout_protocol`, `recursive_workday`, `evacuation_drill_loop`
- 피한다: 실제 회사/팀/건물/내부망 명칭이 들어간 id

## 새 타입 후보 백로그

| 우선순위 | 후보 id | 한 줄 컨셉 | 플레이 차이 | 첫 구현 단위 |
|---:|---|---|---|---|
| 1 | `blackout_protocol` | 전력 복구 절차가 사람보다 설비 우선순위를 지키며 사무실을 어둡게 재배치한다. | 배터리/조명/보안문 압박이 강해지고 탕비실·서버실보다 비상계단·배전함 단서가 중요해진다. | 배터리 조건 인카운터 1개, 비상등 로그, 정전 탈출 엔딩 변주 |
| 2 | `recursive_workday` | 같은 업무일이 다른 로그 번호로 반복되고, 플레이어의 이전 선택이 사내 기록처럼 남는다. | 정신력과 로그 해석이 핵심이며, 반복 플래그가 선택지를 열거나 오염시킨다. | 반복 회의/메일 인카운터 1개, 로그 상세 패널 문구 변주 |
| 3 | `evacuation_drill_loop` | 대피 훈련 방송이 실제 대피를 무한히 리허설로 되돌린다. | 이동 경로와 위험도 관리가 중심이며, 로비/비상계단/방송실 루트가 바뀐다. | 방송실 선택지와 로비 탈출 엔딩 변주 |
| 4 | `audit_ghost_shift` | 감사 시스템이 퇴근하지 못한 사람들의 업무 흔적을 감시 항목으로 되살린다. | 업적/증거/진실 공개 루트와 잘 맞고, 선택 실패가 기록 오염으로 남는다. | 증거 보관/삭제 선택지 1개, 진실 엔딩 후일담 변주 |
| 5 | `mirror_office_merge` | 다른 층 또는 다른 회사처럼 보이는 사무실이 현재 사무실과 병합된다. | 위치 이름과 연결이 흔들리며, storypack 확장과 결합하기 좋다. | 위치 설명 변주와 안전한 공개 힌트 1개 |

백로그 후보는 바로 런타임 콘텐츠로 확정하지 않는다.
각 후보는 먼저 공개-safe 검토를 통과한 뒤 `src/tui_adv/data/*.yaml`에 들어간다.

## 후보 승격 기준

후보를 실제 구현 slice로 승격하려면 다음을 모두 만족해야 한다.

- 기본 타입 `unknown_isolation`과 감정/플레이 차이가 분명하다.
- 실제 회사 정보나 민감한 운영 절차를 연상시키지 않는다.
- 기존 자원 5종과 위치 태그만으로 첫 slice를 표현할 수 있다.
- 최소 하나의 smoke 경로가 deterministic seed로 검증 가능하다.
- Python YAML, browser generated JSON, Rust content bundle 중 영향을 받는 산출물을 함께 갱신할 수 있다.
- Textual 시작 UX가 없더라도 CLI/테스트에서 타입 id를 주입할 방법이 있다.

## 기존 문서와의 관계

- `docs/story/Disaster_Cause.md`는 현재 기본 타입 `unknown_isolation`의 진실 루트 문서다.
- 이 문서는 여러 타입을 추가할 때의 규칙과 후보 백로그다.
- `docs/dev/Data_Schema.md`는 `conditions.disaster_types`의 실제 YAML 스키마를 기록한다.
- `docs/dev/TUI_Layout.md` 또는 별도 UI 문서는 시작 화면에서 타입 선택이 필요해질 때 갱신한다.
