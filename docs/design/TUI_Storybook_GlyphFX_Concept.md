# TUI Storybook + GlyphFX Concept v2

Project working title: **Escape from the Office**

Concept version: **v2 — text-first illustrated TUI adventure**

## 1. 한 줄 정의

겉으로는 정적인 텍스트 어드벤처/스토리북 UI처럼 보이지만, 특정 사건이 발생하면 글자, 문단, 선택지, 그림 일부가 살아 움직이는 TUI풍 인터랙티브 게임이다.

이 게임은 실제 Linux 터미널을 재현하는 게임이 아니다. 플레이어에게 `cd`, `ls`, `pwd`, `grep` 같은 명령어를 익히게 하는 것이 목표가 아니다.

핵심은 다음 조합이다.

```text
텍스트 중심의 화면
+ 가끔 등장하는 TUI/픽셀풍 상황 그림
+ 선택지 기반 진행
+ 특정 순간에만 폭발하는 글자/레이아웃 특수효과
```

짧게 말하면, 이 게임은 터미널을 사용하는 게임이 아니라 **텍스트로 이루어진 세계가 때때로 살아 움직이는 게임**이다.

## 2. 방향 수정: 터미널 게임이 아니라 TUI풍 스토리 화면

이전의 `fake terminal`, `panel`, `mail queue`, `file viewer` 같은 단어는 유지할 수 있지만, 지금의 중심 방향은 실제 터미널 조작이 아니다.

### 피해야 할 방향

- 실제 터미널 시뮬레이터
- 사내 OS 조작 게임
- 명령어 입력 중심 해킹 게임
- 검은 배경 + 초록 글자만으로 구성된 UI

### 지향할 방향

- 모바일 텍스트 어드벤처
- 픽셀/ASCII/TUI 감성의 스토리북 화면
- 대화와 상황 설명이 중심인 게임
- 필요할 때만 그림이 들어가는 사건 페이지
- 글자와 레이아웃이 연출 재료가 되는 게임

레퍼런스에서 중요한 것은 RPG 스탯이나 하단 메뉴 자체가 아니라 화면 구조다.

```text
상단: 캐릭터/상태/챕터 정보
중앙: 대화 또는 긴 상황 설명
특정 장면: 상황 그림 또는 픽셀풍 장면 컷
하단: 선택지, 계속하기, 기록/가방/지도 같은 보조 메뉴
```

## 3. 게임의 정체성

이 게임의 정체성은 **텍스트가 세계의 표면**이라는 데 있다.

플레이어는 3D 공간을 직접 돌아다니기보다 문장, 선택지, 그림 조각을 통해 상황을 읽고 판단한다. 평상시 화면은 안정적인 텍스트 페이지처럼 보인다. 그러나 이상 현상, 기억 오류, 감시, 금지된 문서, 탈출 단서가 등장하면 화면을 구성하던 글자들이 물리적으로 반응한다.

가능한 반응 예시:

- 문장이 순간적으로 깨진다.
- 특정 단어만 제자리에 남고 나머지가 흩어진다.
- 선택지가 나타나려다가 지워진다.
- 그림 속 모니터의 글자가 본문 영역으로 흘러나온다.
- 보이지 않는 물체가 지나가며 문단을 밀어낸다.
- 거짓 문장은 흔들리고, 진짜 단서는 안정적으로 남는다.

TUI 감성은 단순한 스킨이 아니다. 글자, 선, 여백, 선택지, 그림 컷이 모두 게임 세계의 물질이다.

## 4. 기본 화면 모델

### 4.1 텍스트 중심 화면

가장 기본이 되는 화면이다. 그림 없이 대화와 상황 설명만 표시된다.

```text
┌────────────────────────────────────┐
│ [초상화]  이름 / 상태 / 챕터       │
├────────────────────────────────────┤
│                                    │
│  당신은 끝없는 회의와 보고서의     │
│  미로 속에서 정신을 차렸다.        │
│                                    │
│  복사기의 낮은 진동이 복도 끝에서  │
│  울려 퍼지고, 창밖의 도시 불빛은   │
│  오래된 모니터처럼 깜빡인다.       │
│                                    │
│  동료:                             │
│  "정말 가려는 거야?"              │
│                                    │
├────────────────────────────────────┤
│  1. 사원증을 숨긴다                │
│  2. 책상 아래를 살핀다             │
│  3. 아직 움직이지 않는다           │
└────────────────────────────────────┘
```

핵심은 읽기 좋은 텍스트, 충분한 여백, 차분한 정적 분위기다.

### 4.2 그림이 있는 상황 화면

특정 사건, 장소 발견, 중요한 선택 전후에는 그림이 들어갈 수 있다.

```text
┌────────────────────────────────────┐
│ [초상화]  이름 / 상태 / 챕터       │
├────────────────────────────────────┤
│                                    │
│        [ TUI/픽셀풍 장면 그림 ]     │
│        낡은 기록보관실 / 모니터    │
│                                    │
├────────────────────────────────────┤
│  안개 낀 복도를 따라 걷다 보니,    │
│  낡은 나무 문 하나가 희미하게      │
│  빛나고 있었다.                    │
│                                    │
│  문 위에는 작은 황동판이 붙어 있다.│
│  [ 인사부 · 기록보관팀 ]           │
│                                    │
├────────────────────────────────────┤
│  ✥ 의지서를 책상 위에 올려놓는다   │
│  ✥ 아직 때가 아니라고 판단한다     │
└────────────────────────────────────┘
```

그림은 매 화면마다 필요하지 않다. 오히려 가끔 등장해야 의미가 커진다.

### 4.3 효과가 발생하는 화면

평소에는 정적인 텍스트 페이지였던 화면이 특정 순간에만 변한다.

```text
정상 상태:
  문단은 정돈되어 있음
  선택지는 안정적으로 보임
  그림은 배경처럼 존재함

이상 상태:
  문장 일부가 흔들림
  특정 단어가 다른 문자로 치환됨
  그림 속 텍스트가 본문 위로 흘러나옴
  선택지가 잠깐 사라졌다가 복구됨

복구 후:
  단서가 되는 단어만 선명하게 남음
```

## 5. UI 구성 요소

### 5.1 상단 정보 영역

상단에는 캐릭터 초상화, 이름, 챕터, 간단한 상태 정보가 들어갈 수 있다. 그러나 필수 HUD라기보다 페이지의 장식적 문맥에 가깝게 다룬다.

사용 가능:

- 현재 인물 이름
- 현재 장소
- 챕터/페이지 번호
- 간단한 심리 상태
- 중요한 리소스 1~2개

피해야 할 것:

- 항상 많은 스탯을 노출
- RPG 전투 UI처럼 복잡한 수치 표시
- 게임 메커닉과 연결되지 않는 장식성 아이콘

### 5.2 본문 영역

가장 중요한 영역이다. 본문은 항상 읽기 좋아야 한다.

역할:

- 대화
- 상황 설명
- 내면 독백
- 문서/메모/기록
- 선택 전후 결과

GlyphFX가 있더라도 기본 텍스트 가독성을 해치면 안 된다.

### 5.3 그림 영역

그림은 다음 상황에서만 사용한다.

- 새로운 장소 발견
- 중요한 인물과 조우
- 위험한 물체 발견
- 선택의 무게를 보여줘야 할 때
- 텍스트만으로 분위기 전달이 부족할 때

스타일은 고해상도 일러스트보다 픽셀풍, TUI풍, ASCII적 질감이 어울린다. 반드시 순수 ASCII일 필요는 없지만 화면 전체의 텍스트성과 잘 맞아야 한다.

### 5.4 선택지 영역

선택지는 명령어가 아니라 서술형 행동이다.

좋은 선택지:

```text
1. 사원증을 숨기고 비상구로 향한다.
2. 동료의 책상에서 힌트를 찾아본다.
3. 아직 때가 아니다. 자리에 앉는다.
```

피해야 할 선택지:

```text
> cd emergency_exit
> ls desk
> grep clue
```

명령어풍 표현은 특별한 장면의 연출로는 쓸 수 있지만, 기본 조작 방식이 되어서는 안 된다.

### 5.5 하단 메뉴

하단 메뉴는 선택 사항이다. 현재 게임에 꼭 필요한 것만 남긴다.

후보:

- 기록
- 소지품
- 단서
- 인물
- 지도
- 설정

우선순위 낮음:

- 업적
- 상점
- 많은 수집형 메뉴

## 6. Pretext에 끌린 진짜 이유

Pretext에 대한 관심은 텍스트 줄바꿈을 편하게 하려는 것이 아니다. 중요한 것은 다음 구조다.

```text
1. 텍스트를 미리 분석한다.
2. 글자/단어/줄/문단의 위치와 폭을 알 수 있다.
3. 배치 결과를 재사용한다.
4. 특정 상황에서 텍스트의 흐름과 위치를 바꿀 수 있다.
5. 글자 하나하나를 연출 대상으로 삼을 수 있다.
```

이 사고방식을 이 프로젝트에서는 GlyphFX로 확장한다.

```text
Pretext-like layout:
  문단을 어디에 어떻게 배치할지 계산한다.

GlyphFX:
  배치된 글자를 효과의 재료로 쓴다.
```

## 7. GlyphFX 정의

GlyphFX는 글자, 단어, 문장, 선택지, 장식선, 그림 속 텍스트를 움직이는 물질처럼 다루는 연출 시스템이다.

### 7.1 기본 데이터

```ts
type Glyph = {
  id: string

  char: string
  originalChar: string

  lineIndex: number
  charIndex: number

  baseX: number
  baseY: number
  x: number
  y: number

  width: number
  height: number

  opacity: number
  color: string
  styleTag?: string

  semanticTag?: string
  seed: number
  corruption: number
}
```

핵심은 `baseX/baseY`와 `x/y`를 분리하는 것이다.

```text
baseX/baseY:
  정적인 정상 위치

x/y:
  현재 효과가 적용된 실제 위치
```

이 구조 덕분에 화면은 평소에는 안정적이고, 사건이 발생하면 글자들이 일시적으로 움직인 뒤 다시 원래 자리로 돌아올 수 있다.

### 7.2 semanticTag

GlyphFX는 랜덤 장식이 아니라 이야기와 연결되어야 한다.

semanticTag 예시:

- `exit-clue`
- `forbidden-document`
- `manager-lie`
- `memory-corruption`
- `security-scan`
- `hidden-choice`
- `office-noise`

예를 들어 `exit-clue` 태그가 붙은 단어는 글리치가 발생해도 끝까지 안정적으로 남을 수 있다. 반대로 `manager-lie` 태그가 붙은 문장은 읽는 동안 조금씩 흔들리거나 잘못된 문자로 바뀔 수 있다.

## 8. 효과 분류

### 8.1 정적 페이지 위의 미세 효과

가장 자주 쓰는 효과다. 레이아웃은 유지하고 글자만 조금 반응한다.

- 글자 미세 흔들림
- 단어 깜빡임
- 문장 일부 흐려짐
- 특정 이름만 색이 변함
- 선택지 테두리가 순간적으로 깨짐
- 그림 속 모니터 글자가 깜빡임

사용처:

- 불안감
- 거짓말
- 감시
- 기억 오류
- 선택 직전의 긴장

### 8.2 글리치/치환 효과

문자가 다른 문자로 바뀌거나 읽히지 않는 기호로 잠깐 변한다.

```text
원문:
  다른 길은 언제나 존재한다.

글리치 중:
  다▒ 길은 언□나 존§한다.

복구 후:
  다른 길은 언제나 존재한다.
```

규칙:

- 단서를 완전히 읽지 못하게 하면 안 된다.
- 너무 자주 쓰면 싸구려 해킹 연출처럼 보인다.
- 글리치는 반드시 의미가 있어야 한다.

### 8.3 물체와 글자의 상호작용

움직이는 물체가 글자를 밀거나 가린다.

예시:

- 복사기 소음 파동이 문장을 밀어냄
- 보안 스캔 라인이 단어를 지움
- 그림 속 커서가 본문으로 튀어나옴
- 검은 사각형이 선택지를 지나가며 일부를 가림
- 모니터의 초록 글자가 화면 아래로 흘러나옴

이 효과는 텍스트의 위치를 알고 있어야 특정 물체와 충돌시키거나 회피시킬 수 있기 때문에 Pretext-like 기법과 잘 맞는다.

### 8.4 진짜 리플로우 효과

특별한 장면에서만 사용한다. 문단 자체가 다시 흘러간다.

예시:

- 본문 중앙에 금지된 도장이 찍히고 문장이 그 주변으로 밀려남
- 그림 속 인물이 나타나며 대화 문단이 좌우로 갈라짐
- 선택지가 본문 속에서 솟아나오듯 재배치됨
- 문서가 살아있는 것처럼 줄바꿈을 바꾸며 단서를 드러냄

이 효과는 강력하지만 남발하면 UI가 불안정해 보인다. 보스전, 중요한 기억, 탈출 단서 발견처럼 큰 순간에만 사용한다.

## 9. 화면 톤

레퍼런스 이미지가 판타지 RPG처럼 보일 수 있어도, 실제 톤은 다음에 가깝다.

```text
사무실
야근
기록보관실
복사기
결재 문서
사원증
회의실
퇴근 불가
낡은 모니터
비상구
감시 시스템
초현실적 관료주의
```

좋은 무드:

- 따뜻하지만 답답한 종이색
- 오래된 업무 문서
- 픽셀풍 장식선
- 도장, 결재란, 사원증, 복사 흔적
- 초록빛 모니터 텍스트
- 고장 난 형광등
- 정돈된 페이지가 서서히 이상해지는 느낌

피할 무드:

- 순수 중세 판타지
- 네온 해커 터미널
- 과도한 RPG 스탯 게임
- 복잡한 전투 UI

## 10. 플레이 방식

플레이어의 기본 행동은 읽고, 선택하고, 조사하는 것이다.

기본 동사:

- 계속하기
- 선택하기
- 조사하기
- 대화하기
- 기억하기
- 숨기기
- 사용하기
- 기다리기
- 돌아서기

터미널 명령어가 아니라 상황에 맞는 행동 문장이 중심이다.

좋은 예:

```text
의지서를 책상 위에 올려놓는다.
동료의 책상에서 힌트를 찾아본다.
사원증을 안주머니에 숨긴다.
복도 끝의 붉은 불빛을 따라간다.
```

피해야 할 예:

```text
use badge
open file
cd hallway
ls archive
```

특수한 시스템 화면에서는 `open`, `archive`, `restore` 같은 단어를 UI 장식으로 사용할 수 있다. 하지만 플레이어가 터미널 문법을 알아야 하는 구조는 피한다.

## 11. 구현 방향

### 11.1 런타임

추천 방향은 웹 기반이다.

```text
Primary:
  Web / mobile portrait first

Later:
  Tauri or Electron desktop build

Not primary:
  native Linux terminal app
```

이유:

- 일반 게이머 접근성이 좋다.
- 모바일 세로형 UI와 잘 맞는다.
- 폰트, 그림, 애니메이션 제어가 쉽다.
- GlyphFX 구현이 훨씬 자유롭다.
- 실제 터미널 호환성 문제를 피할 수 있다.

### 11.2 렌더링 방식

하이브리드 렌더링이 가장 현실적이다.

```text
Static UI:
  HTML/CSS
  읽기 좋은 본문, 선택지, 버튼, 메뉴

GlyphFX Layer:
  Canvas overlay
  글자 흔들림, 치환, 밀림, 파편화, 리플로우 효과

Illustration:
  이미지 또는 Canvas
  픽셀풍 장면 컷, 그림 속 텍스트 애니메이션
```

모든 글자를 처음부터 Canvas로만 그릴 필요는 없다. 일반 상태에서는 HTML/CSS가 안정적이다. 효과가 발생하는 순간에는 해당 텍스트 블록을 GlyphFX 레이어로 넘겨 글자별 애니메이션을 처리한다.

### 11.3 작동 예시

```text
1. 평상시:
   HTML로 본문과 선택지를 렌더링한다.

2. 이상 현상 시작:
   현재 본문 텍스트를 prepare한다.
   글자별 위치와 줄 정보를 계산한다.
   HTML 본문을 잠깐 투명하게 하거나 마스킹한다.
   Canvas GlyphFX 레이어에서 같은 글자를 그린다.

3. 효과 진행:
   글자 흔들림, 치환, 밀림, 복구를 수행한다.

4. 효과 종료:
   Canvas 레이어를 제거한다.
   HTML 본문을 다시 보여준다.
   단서가 되는 단어만 강조 상태로 남긴다.
```

## 12. Pretext의 역할

Pretext 또는 Pretext-like 엔진은 다음 역할에 적합하다.

- 문단의 줄바꿈 계산
- 텍스트 블록의 라인 정보 계산
- 글자/단어 단위 위치 계산의 기반
- 리사이즈 시 layout 재계산
- 리플로우 효과를 위한 사전 준비

역할 분담:

```text
Pretext:
  layout planner

GlyphFX:
  animation/effect system

GameCore:
  story state and choices

Renderer:
  page UI and visual output
```

Pretext는 게임 그 자체가 아니라 텍스트를 연출 가능한 상태로 만드는 기술적 기반이다.

## 13. MVP 제안

처음부터 많은 메뉴와 스탯을 만들 필요는 없다. MVP는 세 화면이면 충분하다.

### 화면 1: 텍스트 중심 장면

```text
장소:
  야근 중인 사무실

내용:
  플레이어가 낡은 사원증을 발견한다.

필요 요소:
  상단 이름/챕터
  본문 텍스트
  3개 선택지
  하단 기록/소지품 정도의 간단 메뉴
```

### 화면 2: 그림이 있는 상황 장면

```text
장소:
  기록보관팀 앞 복도

내용:
  문 앞 황동판과 초록빛 모니터를 발견한다.

필요 요소:
  픽셀/TUI풍 장면 그림
  상황 설명
  2개 선택지
```

### 화면 3: GlyphFX가 발생하는 장면

```text
장소:
  기록보관팀 내부

내용:
  모니터의 문장이 본문으로 흘러나와 선택지를 방해한다.

필요 효과:
  글자 깜빡임
  단어 치환
  보이지 않는 물체가 문단을 밀어내는 효과
  효과 후 단서 단어만 안정적으로 남음
```

MVP에서 만들지 않아도 되는 것:

- 복잡한 스탯 시스템
- 상점
- 업적
- 전투
- 실제 터미널 입력기
- 완전한 인벤토리 시스템
- 모든 화면에서 발생하는 화려한 애니메이션

## 14. 샘플 장면 문안

### 14.1 텍스트 중심 장면

```text
당신은 끝없는 회의와 보고서의 미로 속에서 정신을 차렸다.

탁한 형광등이 깜빡이고, 복사기의 저음이 고요한 복도에 울려 퍼진다.
오늘도 야근 확정.

그때, 책상 서랍 안쪽에서 낡은 사원증 한 장이 손끝에 닿았다.
뒷면에는 흐릿한 글씨로 적혀 있다.

'다른 길은 언제나 존재한다.'

창밖에 먹구름이 몰려온다.
지금이 기회일지도 모른다.
```

선택지:

```text
1. 사원증을 움켜쥐고 비상구로 향한다.
2. 동료의 책상에서 힌트를 찾아본다.
3. 아직 때가 아니다. 자리에 앉는다.
```

### 14.2 그림이 있는 상황 장면

```text
안개 낀 복도를 따라 걷다 보니, 낡은 나무 문 하나가 희미하게 빛나고 있었다.

문 위에는 작은 황동판이 붙어 있다.

[ 인사부 · 기록보관팀 ]

문을 열자 오래된 종이와 잉크 냄새가 코를 찔렀다.
방 안에는 단 하나의 책상이 놓여 있고,
그 위의 모니터에는 초록빛 글자가 깜빡이고 있다.

"여기에 올려놓으십시오.
당신의 의지서가 다음 여정의 기준이 됩니다."
```

선택지:

```text
✥ 의지서를 책상 위에 올려놓는다.
✥ 아직 때가 아니라고 판단하고 돌아선다.
```

### 14.3 GlyphFX 장면

```text
모니터의 초록 글자가 갑자기 본문 위로 번졌다.

처음에는 한 줄이었다.
곧 문장이 되었다.
그리고 문장은, 당신이 읽고 있던 문단을 밀어내기 시작했다.

[ 승인되지 않은 퇴근 시도 ]
[ 사유를 제출하십시오 ]
[ 사유를 제출하십시오 ]
[ 사유를 제출하십시오 ]

문장들이 서로 겹치고 깨지는 사이,
단 하나의 단어만 이상할 정도로 선명하게 남았다.

비상계단.
```

효과:

- `사유를 제출하십시오`가 반복되며 본문을 침범
- 선택지 일부가 깨짐
- `비상계단`만 안정적인 글자로 남음
- 플레이어는 이 단어가 단서임을 인식

## 15. 제작 원칙

### 15.1 정적 화면이 기본이다

이 게임은 항상 움직이는 UI가 아니다. 평상시에는 차분하고 읽기 쉬워야 한다.

```text
기본 상태:
  정돈됨
  조용함
  여백이 있음
  문장이 잘 읽힘

특수 상태:
  잠깐 깨짐
  의미 있게 흔들림
  사건이 끝나면 복구됨
```

### 15.2 효과는 의미가 있어야 한다

모든 글리치 효과는 다음 중 하나의 목적을 가져야 한다.

- 단서 공개
- 위협 표현
- 거짓말 암시
- 기억 오류 표현
- 선택지의 위험성 전달
- 세계의 규칙 설명
- 특정 존재가 화면에 개입했다는 증거

의미 없는 효과는 제거한다.

### 15.3 UI는 과하게 게임화하지 않는다

이 게임의 핵심은 텍스트와 선택이다. 스탯이 들어가더라도 적게, 명확하게, 서사적으로 써야 한다.

좋은 예:

```text
의지 3
피로 7
감시도 2
남은 시간 18:42
```

나쁜 예:

```text
공격력, 방어력, 민첩, 행운, 마나, 골드, 경험치 등 너무 많은 RPG 수치
```

### 15.4 그림은 사건의 밀도를 높일 때만 쓴다

매 장면에 그림을 넣으면 텍스트 어드벤처의 긴장감이 흐려질 수 있다. 그림은 중요한 장소, 인물, 물체, 선택 직전에 사용한다.

## 16. 최종 결정

현재 아이디어의 가장 적절한 정의는 다음이다.

```text
장르:
  TUI풍 텍스트 어드벤처 / 인터랙티브 스토리 게임

화면:
  모바일 세로형 스토리 페이지
  대화와 상황 설명 중심
  필요할 때만 픽셀/TUI풍 그림 삽입

조작:
  선택지 기반
  조사/사용/대화/기억하기 중심
  실제 shell command는 사용하지 않음

기술:
  Web first
  HTML/CSS static story UI
  Canvas GlyphFX overlay
  Pretext-like text prepare/layout/cache

정체성:
  정적인 문서/업무 페이지가 특정 순간 살아 움직이는 게임
```

Prototype target:
