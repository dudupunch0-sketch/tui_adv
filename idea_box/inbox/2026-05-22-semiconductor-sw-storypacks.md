# 반도체 대기업 SW 중심 스토리팩 후보

## 원문/요지

사용자는 `idea_box/semiconductor_sw_storypacks.md`에 반도체 대기업의 SW 개발센터를 배경으로 한 대형 스토리팩 시나리오 문서를 추가했다.

원문은 특정 팩 하나가 아니라, 공통 회사 배경과 반복 등장인물, 그리고 서로 연결 가능한 4개의 스토리팩을 제안한다.

```text
공통 톤: 블랙코미디 / 코스믹호러 / 평범한 대기업 재난극
주무대: 팹 내부가 아니라 SW 개발센터, 사무동, 서버실, 회의실, 사내망, 개발/인사/보상 시스템
핵심 공포: 괴물보다 회의록, 빌드 로그, 사양서, 권한, 평가, 근태, 연봉표 같은 평범한 회사 시스템
```

이 문서는 원문을 나중의 설계/개발 agent가 검토할 수 있도록 구조화한 아이디어 후보 문서다. 아직 확정 세계관이나 구현 요구사항이 아니며, 기존 게임 스토리/런타임 콘텐츠를 바로 대체하라는 뜻도 아니다.

## 핵심

이 아이디어의 중심은 “회사는 끝까지 평범해야 한다”는 점이다.

회사가 비밀 연구소이거나 악마적 조직인 것이 아니라, 아주 정상적인 대기업 시스템이 비정상적인 사건 이후에도 너무 정상적으로 돌아가기 때문에 공포가 생긴다.

핵심 문법:

- 이상현상은 초자연적으로 보이지만, 회사 시스템은 그것을 업무 프로세스로 처리한다.
- 현실이 깨져도 회의는 잡히고, 문서는 공유되고, 빌드는 돌고, 권한은 막히고, 근태 알림은 발송된다.
- 사람은 이름보다 담당자, 오너, 리소스, 비용, 권한, 평가등급, 참석자로 먼저 취급된다.
- 코스믹호러는 괴물의 등장보다 “기록과 시스템이 현실을 재정의하는 순간”에서 나온다.
- 블랙코미디는 말이 안 되는 상황에서도 회사가 회사다운 답변을 하는 데서 나온다.

## 기대 효과

- 기존 `회사-아포칼립스 선택지 게임`의 톤과 매우 잘 맞는 장기 콘텐츠 후보가 된다.
- SW 개발자/대기업 근무자가 공감할 수 있는 소재가 많다.
- `TUI/fake-terminal` 정체성과 잘 맞는다. 메신저, 회의록, 빌드 로그, CI 대시보드, 이슈트래커, 권한 오류, 문서관리 시스템을 UI/연출로 활용할 수 있다.
- 스토리팩별로 독립 플레이가 가능하면서도, 반복 등장인물과 교차 단서를 통해 캠페인식 연결감을 줄 수 있다.
- 런타임 콘텐츠로 바로 넣지 않더라도, 장소/인물/이벤트/엔딩/EffectCue/텍스트 연출의 재료 풀로 사용할 수 있다.

## 공통 배경 후보

### 임시 회사

- 임시 회사명: `한빛세미콘`
- 사업 영역 후보: 메모리, 컨트롤러, 차량용 칩, AI 가속기, 모바일 AP
- 플레이어 주요 접점: 본사 부속 `SW 개발센터`

주의: 회사명과 설정은 후보명이다. 실제 회사/조직/제품을 연상시키는 세부사항은 공개 문서나 런타임 콘텐츠에 넣기 전에 충분히 일반화해야 한다.

## 스토리팩별 구현 후보

### 1. 차원격리팩

구현 후보:

- `#격리자-임시대응방`을 중심 UI/서사 장치로 둔다.
- 플레이어가 직접 볼 수 없는 공간의 단서를 동료 메시지와 로그로 조합한다.
- Git 브랜치/릴리즈 브랜치/세계선 브랜치 개념을 엔딩 분기와 연결한다.
- “체리픽” 같은 개발 용어를 실제 윤리적 선택으로 번역한다.

런타임 후보:

- location: 개발팀 사무공간, 서버실 앞 복도, 검증 모니터링룸, 방재실, 회의실 복도
- flags: `isolation_channel_joined`, `worldline_branch_seen`, `release_branch_conflict_seen`
- possible endings: `current_branch_kept`, `success_worldline_merged`, `person_cherry_picked`, `isolation_logs_disclosed`

### 2. 문서오염팩

구현 후보:

- 사양서 열람 여부에 따라 같은 장소/인물/이슈가 다르게 보이게 한다.
- “열람 금지 공지에 문제 문서가 첨부되는” 식의 블랙코미디를 적극 활용한다.
- 문서가 현실을 설명하는 것이 아니라 정의하는 방향으로 진행한다.
- 읽지 않은 인물은 기준점이지만, 핵심 단서에 접근하지 못하게 만들어 딜레마를 만든다.

런타임 후보:

- items: `contaminated_spec_pdf`, `correction_notice`, `printed_spec_excerpt`
- flags: `spec_opened`, `baseline_reader_preserved`, `spec_reference_spread`, `incompatible_member_seen`
- possible endings: `spec_discarded`, `errata_published`, `spec_approved`, `spec_disclosed`

### 3. 회의실예약팩

구현 후보:

- 회의실 예약 폼의 조건이 곧 퍼즐 입력이 된다.
- 안건/참석자/시간/반복 여부/티켓 번호/브랜치명을 조합해 필요한 회의실로 이동한다.
- 회의록은 단서이면서 현실을 바꾸는 위험한 아이템으로 취급한다.
- “아무도 참석하지 않은 회의가 자동으로 결정을 내림”을 주요 공포로 둔다.

런타임 후보:

- locations: 실패한 프로젝트 리뷰룸, 반복 주간회의실, 미래의 장애 회고실, 취소된 임원 보고실, 참석자 없음 회의실
- mechanics: 예약 조건 기반 이동, 회의록 획득/등록/수정, 참석자 명단 퍼즐
- possible endings: `meeting_closed`, `meeting_cancelled`, `minutes_modified`, `no_attendee_decision`

### 4. 연봉협상-파업팩

구현 후보:

- 파업을 공장 라인이 아니라 SW 개발센터의 멈춤으로 표현한다.
- 이슈트래커, 빌드팜, 리그레션, 온콜, 배포 승인 정지를 중심 사건으로 둔다.
- 보상 모델 수치가 사내 시스템의 권한/기억/기록을 바꾸는 구조로 간다.
- 사측/노조/인사/재무/SW 엔지니어 그룹을 모두 단순 악역으로 만들지 않는다.

런타임 후보:

- items: `compensation_simulation_xlsx`, `leaked_pay_model`, `strike_vote_notice`
- flags: `baseline_below_seen`, `essential_staff_auto_assigned`, `pay_model_disclosed`, `keyboard_down_chosen`
- possible endings: `agreement_reached`, `strike_started`, `management_wins`, `union_wins`, `pay_model_disclosed`, `baseline_destroyed`

주의: 노동/파업/보상 소재는 현실적으로 민감하다. 게임 내 허구의 사건으로 분명히 처리하고, 특정 실제 회사/노조/인물을 연상시키는 세부사항은 피해야 한다.

## 스토리팩 간 연결 후보

원문은 각 팩이 독립 캠페인이면서도 단서/인물/결과가 서로 이어지는 구조를 제안한다.

- 차원격리팩 → 문서오염팩
  - 격리자 채널에서 공유된 회의록 일부가 오염된 사양서에 인용된다.
- 문서오염팩 → 회의실예약팩
  - 오염된 사양서 각주에 존재하지 않는 회의실명이 적혀 있고, 그 회의실을 예약하면 최초 승인 회의로 들어간다.
- 회의실예약팩 → 연봉협상-파업팩
  - 미래 장애 회고실에서 `필수 유지 인력 자동 지정` 같은 문구가 먼저 등장한다.
- 연봉협상-파업팩 → 차원격리팩
  - 보상 모델의 특정 시나리오가 격리자 채널 인물들을 `핵심 인력`, `대체 가능`, `보상 제외`, `존재 보류`로 분류한다.

이 연결은 바로 구현 요구사항이 아니라 장기 캠페인 설계 후보로 보아야 한다.

## TUI/fake-terminal 연출 후보

원문에서 특히 잘 맞는 표현들:

```text
[근태알림] 현재 위치가 확인되지 않습니다. 무단이탈 처리될 수 있습니다.
[회의초대] 릴리즈 리스크 리뷰 / 장소: 현재 존재하지 않는 회의실
Test Case: Reality_Current_Office_Isolation / Status: Flaky
[문서관리 시스템] 해당 문서는 삭제할 수 없습니다. 현재 47개 회의록에서 참조 중입니다.
[자동회의록] 참석자 전원 실종으로 회의가 조기 종료되었습니다.
[급여명세서] 이번 달 지급액: 존재 확인 필요
```

구현 시 활용 가능한 UI 표면:

- 메신저 채널 로그
- CI/test dashboard
- Git branch list
- PDF/문서관리 화면
- 회의실 예약 폼
- 자동 회의록
- 익명 게시판/사내 공지
- 급여명세서/보상 시뮬레이션 테이블
- 권한 오류/VDI 접속 실패 화면

## 데이터화 후보

나중에 실제 콘텐츠로 승격한다면 원문을 한 번에 통째로 구현하지 말고, 다음 단위로 쪼개는 것이 좋다.

- `storypack` 메타데이터: id, 제목, 톤, 핵심 시스템, 위험 주제
- `locations`: 사무공간, 서버실, 회의실, 방재실, 인사팀 회의실 등
- `characters`: 반복 등장인물 후보와 관계/신뢰/의심 상태
- `events`: 메시지, 알림, 문서 열람, 회의실 예약, 빌드 실패, 권한 오류
- `items`: 오염 문서, 회의록, 보상 시뮬레이션, 로그, 아티팩트
- `flags`: 열람 여부, 공유 여부, 세계선 확인, 기준선 분류, 필수 유지 인력 지정
- `endings`: 스토리팩별 결말 후보
- `effect_cues`: fake-terminal glitch, dashboard rewrite, branch list mutation, meeting-room jump 등

## 주의점

- 이 아이디어는 아직 후보이며, 기존 메인 스토리/런타임 데이터를 즉시 대체하지 않는다.
- 원문은 4개 대형 스토리팩을 담고 있으므로 한 PR/한 작업으로 전부 구현하기에는 범위가 크다.
- 반도체 대기업, 보상, 파업, 평가, 권한, 문서 유출 등 현실감이 강한 소재는 실제 회사/직원/내부 정보처럼 보이지 않게 일반화해야 한다.
- 팹/제조 공정 지식을 퍼즐 핵심으로 만들지 않는다. 원문의 핵심은 SW 개발센터의 일상과 시스템이다.
- 노동/파업 소재는 어느 한쪽을 단순 악역으로 만들면 톤이 얕아질 수 있다. 사측/노조/인사/재무/SW 엔지니어 모두 각자의 논리와 공포를 가져야 한다.
- 반복 등장인물은 매력적이지만, 기존 캐릭터 시스템과 충돌할 수 있다. 실제 NPC로 승격하기 전에 이름/역할/스탯/관계망을 정리해야 한다.
- 블랙코미디는 웃기지만 찝찝해야 한다. 농담이 현실 노동 문제를 가볍게 조롱하는 방향으로 흐르면 안 된다.
- 공개 문서나 런타임 콘텐츠에는 실제 사용자의 사적 메모, 실제 회사명, 실제 프로젝트명, 실제 내부 프로세스처럼 보이는 민감정보를 넣지 않는다.

## 설계자에게 물어볼 질문

- 이 4개 스토리팩을 하나의 장기 캠페인으로 볼 것인가, 독립 DLC/데이터팩 후보로 볼 것인가?
- 기존 기본 게임의 회사-아포칼립스 설정과 `한빛세미콘` 배경은 같은 세계인가, 별도 스핀오프인가?
- 반복 등장인물 13명을 실제 NPC 후보로 승격할 것인가, 아니면 스토리팩 설계용 archetype으로만 둘 것인가?
- 이전 아이디어인 6스탯 등장인물 생성기와 연결해 인물 시트를 만들 것인가?
- 이전 아이디어인 꿈 엔딩 분기 후보와 결합할 것인가, 아니면 이 팩들은 현실/격리 사건 중심으로 유지할 것인가?
- 1차 구현 후보를 하나만 고른다면 어느 팩이 가장 적합한가?
  - 후보: TUI와 가장 잘 맞는 `차원격리팩`, 퍼즐성이 강한 `회의실예약팩`, 문서/텍스트 연출이 강한 `문서오염팩`.
- 스토리팩별 결말은 기존 엔딩 타입(`escape`, `truth`, `conquest`, `hidden`, `failure`)에 매핑할 것인가, 새 타입을 추가할 것인가?
- 메신저/문서/회의록/빌드 로그 UI를 실제 게임 내 시스템으로 구현할 것인가, 아니면 서술 텍스트와 EffectCue로만 표현할 것인가?
- 민감한 현실 소재를 다룰 때 공개 문서/런타임 데이터/비공개 로컬 설정의 경계를 어떻게 둘 것인가?
