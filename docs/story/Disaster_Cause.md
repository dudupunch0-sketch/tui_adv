# 재난 원인 문서

이 문서는 `escape from the office`의 진실 루트가 드러내는 재난 원인을 공개 저장소에 안전하게 적는 설계 문서다.
실제 회사명, 실제 시스템 이름, 실제 사무실 보안 구조, 개인 이름은 쓰지 않는다.

## 한 줄 원인

재난의 표면 원인은 정전이나 감염이 아니라, 회사가 사람과 공간을 업무 로그처럼 격리·복구하려고 만든 `격리 프로토콜`이 사무실 전체를 하나의 재시도 가능한 작업 단위로 오인한 것이다.

## 플레이어가 알게 되는 순서

1. 퇴사자의 메시지
   - 사내망 알림이 정상 시간보다 늦게 도착한다.
   - 런타임 단서: `ex_employee_contacted`, `ex_employee_memo`
2. 존재하지 않는 부서의 회의록
   - 회의 시간이 모두 같은 초 단위로 반복된다.
   - 런타임 단서/플래그: `meeting_pattern_noticed`, `impossible_meeting_saved`
3. 지연된 CCTV와 서버 로그
   - CCTV는 플레이어가 방금 한 행동을 이미 업무 로그처럼 보관한다.
   - 런타임 단서/플래그: `server_log_fragment`, `isolation_protocol_revealed`
4. 진실 엔딩
   - 엔딩 id: `truth_isolation_protocol`
   - 결론: 사라진 것은 직원들이 아니라 플레이어가 접속한 업무공간의 기준 좌표다.

## 격리 프로토콜의 규칙

- 회사 시스템은 “업무 중단”을 장애로 취급한다.
- 장애 복구 로직은 마지막 정상 업무 상태를 되살리려 한다.
- 사람의 생존보다 회의, 결재, 출입, 배포 로그의 정합성을 먼저 맞춘다.
- 공간 왜곡은 물리 현상이 아니라 잘못 복구된 업무 상태의 부작용이다.
- 플레이어가 탈출·정복·진실 루트 중 무엇을 선택하느냐에 따라 프로토콜을 끊거나, 덮어쓰거나, 이해한 채 남길 수 있다.

## 공개 저장소 금지 정보

다음 정보는 이 문서와 공개 YAML에 넣지 않는다.

- 실제 회사명, 실제 부서명, 실제 프로젝트명
- 실제 출입 통제 구조, 보안실 위치, 서버실 구조
- 실제 직원 이름, 메신저 계정, 전화번호, 이메일
- 현실 연결 힌트의 최종 좌표나 개인 자리 정보
- 실제 장애 대응 절차를 유추할 수 있는 내부 운영 정보

## 런타임 연결 계약

현재 구현은 새 YAML을 추가하지 않고도 다음 공개-safe hook으로 재난 원인 흐름을 표현한다.

| 단계 | hook | 용도 |
|---|---|---|
| 시작 | `truth_route_started` | 전임자 검색으로 진실 루트 진입 |
| 회의록 | `meeting_pattern_noticed` | 반복 회의 패턴 인지 |
| 보안실 | `server_log_fragment` | 격리 프로토콜 로그 조각 확보 |
| 공개 | `isolation_protocol_revealed` | 재난 원인 공개 상태 |
| 결말 | `truth_isolation_protocol` | 진실 엔딩 판정 |

## 서사 톤

재난 원인은 음모 폭로가 아니라 회사 시스템의 관성에 가깝다.
플레이어는 “누가 악당인가”보다 “사람 없는 회사가 왜 아직 정상 운영을 요구하는가”를 이해하게 된다.

## 다음 구현 후보

- 서버 로그 조각을 더 세분화해 `격리 프로토콜` 문장을 단계적으로 복원한다.
- 진실 엔딩 직전 선택지에 “공개/삭제/보관” 분기를 추가한다.
- 정복 루트에서 같은 원인을 다른 감정으로 읽도록 시스템 제압 문구를 확장한다.
